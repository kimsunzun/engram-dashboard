# ADR — 실행 절차 (범용 골격)

`$ARGUMENTS` = 오퍼레이션 `new`|`supersede`|`lint` (옵션). 미지정이면 요청으로 추정한다. 어느 오퍼레이션을 도는지 호출 시 사용자에게 한 줄 명시한다.

이 파일은 **스택을 모르는 범용 골격**이다 — 오퍼레이션별 "판단 → 스크립트 호출 → prose → 보고" 흐름과 불변식만 담는다. ADR 폴더·파일명 규약·**실제 스크립트 명령**·템플릿 실체·인덱스 위치·상태 범례·코드 앵커 명령 같은 **프로젝트 실체**는 바인딩 파일이 채운다.

## 역할 분담 — 결정적 스크립트 + 얇은 판단 (★핵심 구조)

서기 작업(채번·파일 스캐폴드·인덱스 재생성·supersede 양방향 링크 박기·형식 lint)은 **결정적 스크립트**가 한다 — LLM이 손으로 하면 한쪽 빠뜨리고 번호 충돌하고 거짓 lint를 낸다. 스킬(LLM)은 **기계가 못 하는 일**만 한다: ① 입력 수령·검증 ② **전체 폐기 vs 부분 폐기 판단** ③ 본문 prose(맥락·결정·거부한 대안·근거·영향) 채우기 ④ 결과 보고. 경계가 곧 이 스킬의 존재 이유다.

> **바인딩(1회 선언):** 프로젝트 전용 스크립트 명령·경로·템플릿·인덱스 형식·앵커 명령은 `bindings/<project>.md`가 정본이다. 현재 engram = `bindings/engram.md`. 아래 절들이 "스크립트"·"ADR 폴더"·"템플릿"이라 부르는 자리는 그 바인딩에서 실명령·실체를 꺼낸다. 골격은 명령·경로를 하드코딩하지 않는다.

## 0. 인자 파싱·기본값 (오퍼레이션 추정 규칙)

`/adr [new|supersede|lint]` — 오퍼레이션은 옵션이다. ADR 기록은 결정적 작업이라 강도 축이 없다 — 유일한 1차 축이 오퍼레이션이다. 다음 규칙으로만 해석한다(소문자만 허용):

- **정확히 `new`|`supersede`|`lint` 하나** → 해당 오퍼레이션.
- **인자 없음** → 요청 내용으로 추정: 새 결정 박제=`new` / 기존 결정 번복·"이거 폐기"·"바뀜"=`supersede` / "점검"·정합성 확인·인덱스 맞나=`lint`.
- **그 외(알 수 없는 토큰·복수 인자·오타·대문자)** → **실행하지 말고** 받은 인자와 사용법(`/adr [new|supersede|lint]`)을 사용자에게 보고한다(추정 금지).
- **new ↔ supersede가 애매하면**(기존 결정을 *고치는* 건지 *대체*하는 건지 모호) 사용자에게 한 줄 확인한다 — supersede는 옛 결정에 손대므로 임의 추정하지 않는다.

## 1. 입력 수령·검증 (Lead = 메인 오케스트레이터)

오퍼레이션을 돌리기 전에 입력을 갖춘다 — 스킬은 결정을 *만들지 않는다*. 호출자(메인/사용자)가 다음을 제공해야 한다:

- **new** — 결정 제목 · 맥락(무슨 문제) · 결정(무엇으로) · **거부한 대안 + 각 이유** · 근거 · 영향/불변식. 거부한 대안이 비면 호출자에게 되묻는다(ADR의 핵심이 "거부한 대안+이유"라 그게 없으면 다음 세션이 같은 대안을 다시 꺼낸다).
- **supersede** — 위 new 입력 전부 + **어느 기존 ADR을 번복하는지**(번호) + 왜 번복하는지 + **전체 폐기인가 부분 폐기인가**(§2 supersede의 판단).
- **lint** — 입력 없음(점검 전용). 정합성만 본다.

내용의 *타당성*은 판단하지 않는다(검증 상태 참조). 빠진 필수 슬롯만 채워 달라고 되묻고, 채워지면 스크립트 호출로 넘어간다.

## 2. 오퍼레이션별 실행

각 오퍼레이션은 **판단 → 스크립트 호출 → prose 채우기 → (인덱스 재생성)** 순서다. 실제 명령·파일명·템플릿·인덱스 형식은 **바인딩 파일이 정본**이다.

### new — 새 결정 박제

1. **스크립트 호출** — 바인딩의 `new` 명령에 제목(필요시 상태)을 넘긴다. 스크립트가 **채번(기존 최대+1, 쓰기 직전 재스캔)**·파일 스캐폴드·번호 출력을 결정적으로 한다. 번호를 LLM이 임의 지정하지 않는다.
2. **본문 prose 채우기** — 스크립트가 만든 스캐폴드 파일의 빈 슬롯(맥락·결정·거부한 대안·근거·영향)에 §1에서 받은 내용을 채운다. **섹션 구조는 건드리지 않는다**(스크립트/템플릿 정본 그대로). 호출자가 안 준 내용을 지어내지 않는다(§환각 주의).
3. **인덱스 재생성** — 바인딩의 인덱스 재생성 명령을 돌린다(본문 스캔해 표 갱신). 인덱스를 손으로 편집하지 않는다.

### supersede — 결정 번복 (전체/부분 + 양방향)

1. **전체 폐기 vs 부분 폐기 판단 (★LLM의 핵심 판단)** — 옛 ADR이 *통째로* 틀렸나(전체), 아니면 *일부 조항만* 바뀌고 나머지는 살아있나(부분)? 살아있는 조항을 죽이면 안 되므로 **애매하면 사용자에게 확인**한다.
   - **전체(full)** — 옛 결정 전체가 새 결정으로 대체됨. 옛 status를 `폐기`로 바꾸고 양방향 supersede 링크.
   - **부분(partial)** — 옛 결정의 *특정 조항만* 바뀜. **옛 status는 유지**(살아있는 부분이 있으므로 통째 폐기하면 안 됨) + 양방향 `Amends`/`Amended by` 링크에 바뀐 조항을 단서로 박는다.
2. **스크립트 호출** — 바인딩의 `supersede` 명령에 옛 번호 · 모드(full/partial) · (partial이면)바뀐 조항 · 새 제목을 넘긴다. 스크립트가 새 ADR 스캐폴드 + **옛 ADR 양방향 박기**를 결정적으로 한다(full=옛 status→폐기+취소선 보존 / partial=옛 status 유지+관련줄 Amended by).
3. **본문 prose 채우기** — 새 ADR 스캐폴드를 §1 내용으로 채운다(맥락에 "왜 번복하나"). 옛 ADR의 폐기 사유 TODO도 채운다.
4. **인덱스 재생성** — new와 동일.

> **불변식: 양방향 단방향 금지.** 새 ADR에만 "옛 거 대체/개정함"을 적고 끝내면 안 된다 — *옛 ADR만 읽는* 다음 세션이 폐기·개정된 결정을 멀쩡한 줄 알고 따라간다. 스크립트가 양쪽을 박지만, 호출 후 lint로 양방향 일치를 확인한다. (이게 supersede 오퍼레이션의 존재 이유 — 손으로 하면 한쪽을 빠뜨린다.)

> **부분 폐기 = Amends 양방향 링크(개념).** 옛 status를 통째 '폐기'로 덮지 않고, 양방향 `Amends`(새→옛)/`Amended by`(옛→새) 링크 + 바뀐 조항 단서로 *일부만* 바뀌었음을 표시한다. 실명령·실제 링크 문구는 바인딩.

### lint — 정합성 점검 (보고 위주)

1. **스크립트 호출** — 바인딩의 `lint` 명령을 돌린다(read-only). 스크립트가 ① 중복·빠진 번호 ② supersede 양방향 일치(전체·부분) ③ 상태 **어휘만** 유효성(단서절 무시) ④ 코드 앵커 고아(코드 경로 한정, 문서 제외)를 결정적으로 검사해 JSON으로 낸다.
2. **결과 해석·보고** — 스크립트 결과를 사람 말로 옮긴다. **error(진짜 결함)와 advisory(권고·레거시 마이그레이션)를 구분**해 보고한다.

**자동수정 경계:** lint는 **보고 전용**(자동수정 안 함). 인덱스 재생성(`index`)만 본문서 파생 가능한 걸 다시 쓰되, **본문보다 정보가 많은 기존 인덱스 셀(수작업 큐레이션·레거시 부분폐기 단서)은 보존**한다 — 어느 쪽이 정본인지(본문 vs 인덱스)는 맥락 판단이라 자동 손실을 금한다. 의미 있는 drift(양방향 누락·고아 앵커·제목 정본 충돌)는 사용자/메인 판단을 받는다.

## 3. 결과 보고

메인이 오퍼레이션 결과를 사용자에게 보고한다. 스크립트 JSON 출력을 사람 말로 옮긴다.

- **new** — 채번한 번호 · 만든 파일 · prose 채운 섹션 · 인덱스 재생성 결과.
- **supersede** — 모드(전체/부분) · 새 번호 · 번복한 번호 · 양방향 박기 완료(새 ADR + 옛 ADR + 인덱스) · lint로 양방향 확인.
- **lint** — error/advisory 구분한 발견 목록 · 사용자 판단이 필요한 것. 깨끗하면 "정합" 보고.
- 흐름(언제/무엇) 기록·커밋은 **메인이 처리**한다(이 스킬이 직접 쓰지 않는다 — 바인딩 §연동 참조).

## LLM 환각 주의 (★리서치 권고)

LLM이 ADR 본문을 채울 때 **맥락·근거·거부한 대안을 환각**(없는 API·가짜 근거·지어낸 대안)하는 게 검증된 실패 모드다(리서치: Equal Experts "kernel of truth" 패턴). 그래서:

- **거부한 대안·근거·맥락은 호출자(사용자/메인)가 제공한 것만 박제한다.** 빈 슬롯을 그럴듯하게 메우지 않는다 — 비면 되묻는다.
- 채운 본문은 호출자가 **fact-check**하도록 보고한다(스킬이 "옳다"고 보장하지 않음 — 검증 상태 참조).

## 프로젝트 바인딩 (스킬 밖)

이 골격은 **스택을 모르는 범용 ADR 엔진**이다. 프로젝트 전용 실체는 `bindings/<project>.md`가 채운다(현재 engram = `bindings/engram.md`). 바인딩이 정의하는 것:

- **스크립트 실명령** — `new`/`supersede`/`index`/`lint` 각각의 실제 호출 명령(인자 포함).
- **ADR 폴더 + 파일명 규약** — 어디에 어떤 이름으로 파일을 만드나(스크립트가 결정적으로 함 — 바인딩은 그 출처를 짚음).
- **템플릿 실체** — 스캐폴드가 채울 섹션 구조의 정본(가리킴 — 복붙 금지).
- **인덱스 위치·형식 + 재생성 명령** — 표가 어디 어떤 형식인가(정본) + 재생성 명령.
- **상태 범례** — 확정/제안/폐기/거부 등 상태 어휘(스크립트 lint가 이 어휘만 검사).
- **코드 앵커 규약 + 고아 검출** — 코드가 ADR을 어떻게 참조하고(`// ADR-NNNN`), lint이 어느 경로를 긁나.
- **흐름 기록 연동** — 번복·추가를 어느 흐름 로그에 남기나(메인 몫).

다른 프로젝트는 같은 골격에 바인딩 파일만 추가한다. 골격에 특정 경로·명령·템플릿을 하드코딩하지 않는다.

## 가드레일 (앞 절에 없는 금지만)

앞 절에 박힌 규약(채번=스크립트·supersede 양방향·전체/부분 판단·상태 어휘만 검사·인덱스 보존·환각 금지)은 반복하지 않는다. 이 절은 **다른 곳에 안 적힌 금지**만 모은다:

- **결정 날조 금지** — 스킬은 결정 내용·거부한 대안·근거를 *만들지 않는다*. 호출자가 준 것만 박제한다. 빈 필수 슬롯은 채워 달라고 되묻지, 그럴듯하게 지어내지 않는다.
- **ADR 덮어쓰기 금지(누적)** — 기존 ADR 본문을 고쳐 결정을 바꾸지 않는다. 번복은 항상 *새 번호 + 폐기/Amends 표시*(supersede)다. 덮어쓰면 이력이 사라진다.
- **스크립트 우회 금지** — 채번·양방향 링크·인덱스 재생성은 스크립트가 결정적으로 한다. LLM이 손으로 번호를 매기거나 인덱스 표를 직접 편집하지 않는다(그게 1차 초안이 깨진 이유다 — 기계 잡일을 LLM 문서로만 명세).
- **경로·명령 하드코딩 금지** — 골격에 폴더 경로·파일명·스크립트 명령·앵커 명령·템플릿 실체를 박지 않는다. 전부 바인딩이 채운다(이 골격이 스택을 몰라야 다른 프로젝트가 재사용한다).
