# Study Note — 멀티 윈도우 레이아웃 권위/토폴로지 (deep, 2026-06-27)

## 이 리서치가 왜 발동됐나 (메타-교훈)
S14 F1을 메인 세션이 **즉흥 추론**으로 풀려다 두 번 헛디뎠다: ① 처음 "WS 경로 재사용" 편의로 데몬 권위(B안) 밀었다가, ② 사용자가 "레이아웃은 로컬이니 클라가 View 관리해야"라 하자 src-tauri 권위로 *후퇴*. 사용자가 "우리끼리 얘기 vs 전통적으로 쌓인 것은 퀄리티가 다르다 — 리서치 있냐"고 제동. → **굵은 설계를 prior-art 없이 논리로 박으면 안 된다**는 규약(CLAUDE.md 참조구현)의 실증 사례. light로 끝낼 게 아니라 deep이 정당화된 이유 = 되돌리기 비용↑·환각 치명·우리 직관과 관행이 어긋날 징후.

## 쟁점이 어떻게 결론에 이르렀나
- **갈래 설계:** 권위 위치(Q-A)와 토폴로지(Q-B)를 직교 축으로 분리하고, 후보를 3 계열로 클러스터링(터미널 멀티플렉서 / GUI 에디터 / 브라우저·compositor·LSP). by-candidate 팬아웃(§7 설계-결정 모드).
- **핵심 분기 발견:** 터미널 멀티플렉서(서버 권위) ↔ GUI 에디터(창 로컬)가 **정반대**. 이게 쟁점의 심장. 양쪽 다 *왜 그런지*까지 동일하게 진단(내구 객체가 서버측이냐 / UI 그리드를 그 창이 그리냐) → 메타-수렴.
- **결정 갈림:** "engram은 어느 계열인가?" → 터미널 에이전트 멀티플렉서 + 원격 데몬 + §5 LLM 제어 ⇒ **멀티플렉서 계열** ⇒ 서버(데몬) 권위. WezTerm이 *원격 도메인이면 서버 권위 강제*라는 Codex nuance가 쐐기(우리는 항상 데몬 attach=원격형).
- **적대 검증으로 살아남은 반증 2개:**
  1. "레이아웃은 로컬 UI(에디터 관행)" → 반박: 로컬 데몬이면 사실상 로컬 + 우리 계열은 멀티플렉서 + §5는 에디터에 없는 하드 제약.
  2. "에이전트 데몬에 UI 넣는 건 관심사 분리 위반/ADR-0029 정신 위배" → 반박: ADR-0029는 *in-proc 에이전트 호스팅* 금지지 데몬의 세션 데이터 소유 금지가 아님. 오히려 ADR-0029 line40 "데몬=데이터 단일 소유, in-proc 이중소유 제거"가 **src-tauri 권위(b)를 금지**하고 데몬 권위(a)를 지지 — 이 한 줄이 결정적이었다.

## cross-family가 실제로 한 일
사실 차원 불일치 0 — 하지만 Codex가 단독으로 (1) WezTerm local/remote 도메인 nuance, (2) Q-A/Q-B 명시 권고 + 자기반박(중계의 고대역 출력 비용)을 제공해 Claude 팬아웃(prior-art만)보다 *결정 레이어*를 보강. 만장일치였지만 1차 출처(소스/man page)가 받쳐 공통환각 리스크 낮다고 판단.

## deep vs medium 체감
medium이면 갈래 3 + Codex 1회로 끝났을 것. deep이라 Codex 2회(prior-art + 결정권고)로 나눠 *결정 레이어*까지 적대적으로 받았고, 적대 검증에서 ADR-0029 line40 반증-역전이 나왔다. 이 역전이 medium에선 안 나왔을 가능성.
