# 핸드오프: PC 클라 버퍼 = 버그 근본원인 규명 → 제거 재설계 방향 확정 (resync 커밋 23a8c47은 증상대응, 재설계 때 폐기)

## ⭐ 규약 (사용자 결정) ⭐
- 자율 모드에서도 자동 커밋 금지. 명시 지시 때만. 커밋 전 diff 승인. (23a8c47은 "커밋 진행" 명시 승인, push는 안 함.)

## 한 줄 상태 · 다음 첫 액션
resync_output 버그수정을 커밋(23a8c47 로컬)했으나 **증상 대응으로 판명.** **근본원인 = 불필요한 PC 클라 버퍼 + 데몬과의 동기화 복잡도.** 사용자 결정: PC 버퍼 제거(데몬 직수신) + 환경별 분기(모바일만 캐싱 버퍼).
**다음 첫 액션: PC 버퍼 제거 근본 재설계 — 설계부터.** 굵은 설계라 순서 준수: `/research`(OSS 조사·옵션셋) → 사용자 결정 → TRD → ADR-0040 supersede → 구현. 임의 착수 금지.

## ⭐⭐ 핵심 발견 (이번 세션 실질 성과) ⭐⭐
이번 세션 관련 버그 3개가 **전부 "PC 클라 버퍼 ↔ 데몬/화면 동기화" 지점**에서 발생:
1. 리로드 replay 유실(이전 세션 ca3f325) = 프론트 구독 버퍼 flush 레이스
2. split/재배정 remount 고착(이번 23a8c47) = 클라 버퍼 replay 재트리거 누락
3. 다중 slot 공유 유실(B) = 클라 버퍼 진도(lastDeliveredSeq)가 agentId 단위
→ 셋 다 "데몬이 이미 가진 데이터를 클라가 한 벌 더 미러하고 그 둘을 맞추려다" 생기는 동기화 버그. **PC는 데몬이 로컬(loopback 한 홉)이라, 버퍼 없이 remount/재구독 시 데몬에서 직접 받으면 이 계열 버그가 구조적으로 소멸.**

## 재설계 방향 (사용자 결정)
- **PC(로컬):** 클라 버퍼 제거 → remount/mount/재구독 시 데몬에서 직접 replay 수신. src-tauri = 라우팅 pass-through, 프론트 = 화면만.
- **모바일/원격:** 캐싱 버퍼 유지(네트워크 데이터 절감). = **환경별 분기 = transport seam.**
- **ADR-0040("중계 허브 공유 버퍼") supersede 대상.** ADR-0040은 환경별 분기를 "거부한 대안 / 실측 후 후속 별도 ADR로 흡수"로 미뤘고, 그 후속이 안 됨 = 현 버그들의 원인. ADR-0040이 "메모리 중복 ~2MB라 실익 작음"으로 버퍼 유지를 정당화했으나 **실제 비용은 메모리가 아니라 동기화 버그**(이번 세션이 증거).

## resync 커밋(23a8c47) 처리 (사용자 결정 = 1b)
롤백 안 함. **재설계가 클라 버퍼(output_view_store/AgentBufferStore)를 버퍼째 들어낼 때 resync_output도 함께 폐기.** 그때까지 단일 slot remount 버그는 임시 완화로 유지. resync는 증상 대응이라 ADR 불요(step-log만으로 충분할 듯 — 재설계 시 판단).

## 실패한 접근 (do-not — 재시도 금지)
- **resync_output = 증상 대응.** 클라 버퍼 동기화를 한 겹 더 얹은 것. 근본 아님.
- **동기화를 더 정교화하는 방향 금지** — 버퍼 동기화 버그가 또 나오면 답은 "버퍼 제거"지 "동기화 더 정교화"가 아니다.
- getSnapshot(json_agent) tag1 판정 금지(구조화 drop, 항상 0). tag1 판정 = DOM 텍스트 + subs.lastDeliveredSeq.
- subs·pendingBuffers는 JS Map — Object.keys 금지, `[...map.entries()]`.
