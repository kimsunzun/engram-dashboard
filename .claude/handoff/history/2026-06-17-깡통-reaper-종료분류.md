## 3. ★ 이번 세션에서 굳힌 핵심 개념 (재론 금지) ★
- **죽음의 두 축 분리:** **프로세스 종료**(=reaper 대상) vs **대화 세션 회전**(=/clear sid drift, `session_tracker`가 처리, 살아있는 프로세스). 부팅 복원(ADR-0016)은 또 별개 — 런타임 죽음과 섞지 않는다.
- **종료 의도는 "관측"이 아니라 "행동을 일으킨 지점"에서 정한다**(ADR-0019). 유저kill=커맨드 핸들러 태깅 / claude `/exit`·정상=exit code 0 / 크래시=exit≠0 / 데몬셧다운=전역 플래그. PTY 신호만으론 구분 불가(전부 비슷하게 수렴)이므로.
- **cmd /c claude 구조:** master=PTY(conhost) **우리쪽 핸들**(프로세스 아님), child=**cmd.exe**(우리 직속), claude=cmd의 자식(손자). `cmd /c`는 claude 끝날 때까지 안 죽고 claude exit code를 물고 죽음=**투명 대기자** → cmd 핸들 `try_wait`가 곧 claude 죽음+코드. claude는 `.cmd` shim이라 ConPTY가 직접 못 띄움(error 193) → cmd 필수.
- **Windows ConPTY 함정:** master를 쥔 동안 자식이 죽어도 read에 **EOF 안 옴**(conhost가 master 닫기 전까지 파이프 유지). 그래서 자연종료는 read-EOF로 감지 불가 → `child.try_wait` 폴링 watcher 필요(ADR-0016 §37 "child watcher 불필요" 가정이 거짓으로 판명).

## 5. reaper 구현 구조 (`crates/engram-dashboard-core/src/agent/`)
- `reaper.rs`: 단일 supervisor 스레드 + std **unbounded mpsc**. `reap_one`=write lock 안 epoch검증+`sessions.remove()` **Some-승자만** disposition·통지(idempotent), lock 밖 디스크IO/통지(ADR-0006). 루프 **catch_unwind+poison-tolerant**(단일 장애점 격리). `decide()`로 disposition 판정.
- `output_core.rs`: `on_terminal` hook — finalize **승자 1회** 경로에서 intent·shutting_down을 **그 순간 snapshot**해 ReapMsg 발행(reaper live-read race 차단=consult GPT 단독 적출).
- `transport/pty.rs`: **자연종료 watcher** — `child.try_wait` 50ms 폴링 → 자식 종료 시 master drop(`Arc<Mutex<Option>>`) → 기존 EOF→finish→reaper 경로 재사용. **콘솔 detection은 여기에만 격리**(core 중립 — 미래 ApiTransport는 stream-end로 같은 TerminalReason/done만 내면 됨).
- `manager.rs`: kill_agent=intent(UserKill) 태깅 후 shutdown→join_pump(remove/통지는 reaper 위임). shutdown_all=shutting_down **먼저** set 후 kill. spawn=auto_restore=**true 플립을 start_pump 전으로**(reaper downgrade 순서 고정, 크래시 오복원 방지).
