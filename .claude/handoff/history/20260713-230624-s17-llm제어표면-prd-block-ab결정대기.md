## BLOCK 핵심 쟁점 (방향 결함 — 사용자 결정)
**"데몬 opaque-relay(UI payload 해석 안 함)" ↔ "R7 child별 권한 스코프"가 모순.** 권한 검사는 누군가 명령을 읽어야 하는데 데몬은 안 읽기로 함 → `focus`만 허용된 토큰이 opaque 봉투에 `move`를 넣어도 데몬이 못 막음. 이 모순이 ADR-0080의 "UI용 앱-소유 엔드포인트 거부" 근거도 약화(앱은 어차피 UI를 파싱하니 스코프 직접 강제 가능).

갈림길:
- **(a)** 데몬 relay 유지 + 검사 이원화: 데몬=coarse principal 게이트+발신자 도장(semantics 무지 유지), 앱=fine 스코프 검사. 모바일 이식성 유지, 배선 신설 필요.
- **(b)** UI는 클라이언트(앱) 직접 엔드포인트: 읽기+검사 한 곳, 단 별도 엔드포인트·인증, 모바일(원격 데몬) 부적합.
- **(lean, 내 추천)** 권한검사는 항상 권위 소유자(클라이언트 ViewManager)가, 데몬은 UI 의미 무지 유지하되 "인증된 발신자 도장"만 찍기 = (a)+(b) 장점 병합. "opaque"를 *UI semantics 무지*로 좁혀 재정의.

## /review prd FIX 목록 (방향 정해지면 PRD·AC에 반영)
- **카탈로그 범위 vs §5:** §5는 테마·레이아웃 저장/복원·**diff accept/revert**·트리 이동까지 요구. PRD는 `split/tab`만 AC에 두고 "카탈로그 범위"를 통째 TRD로 미룸 → MVP/deferred 선을 PRD에서 명시(안 그러면 §5 갭 은폐). diff accept/revert는 도메인/표현 분류조차 안 됨.
- **상태 조회 primitive 부재(R5):** 관찰이 agent/event만. 클라이언트 view-tree 현재 상태를 읽는 primitive 없음 → "빈 슬롯에 넣어줘" 같은 의미적 상대연산 불가. + 안정 view/tab ID(의미적 주소지정)·stale-ref 미정.
- **R7이 AC에 없음:** 정식 요구인데 수용기준 0 → MVP인지 deferred인지 자기모순. stale-token 수명(kill·재스폰·데몬 재시작) 미정.
- **결과 계약 검증 공백(R6):** 멱등성(재시도 이중적용), 부분적용 원자성, 동시성/순서, `APP_OFFLINE`↔timeout 구분, mixed-version 협상 — AC·미결 어디에도 없음(일부는 TRD-level이나 최소 미결로 노출 필요).

## 용어 (사용자 지적 — 앞으로 지킬 것)
