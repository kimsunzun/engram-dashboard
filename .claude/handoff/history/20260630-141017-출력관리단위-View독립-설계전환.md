## 한 줄 상태 + 다음 첫 액션
박막화(②(a))에 들어가려다, 그 밑에 깔린 **출력 관리 "단위"**(에이전트당 묶음 vs View별 독립) 미결정(study-note N1)이 모든 어긋남의 근원임이 드러났다. 사용자가 방향을 **View별 독립**으로 확정: *"클라 중계 허브(src-tauri)가 에이전트당 공유 버퍼 하나를 갖고, 각 View는 그 버퍼를 가리키는 인덱스(cursor)만 독립."* medium 리서치로 이게 **VS Code pty host + Kafka/Redis offset 모델**과 일치함을 검증했다.

**다음 첫 액션: 이 방향을 정식 설계로 박기 — PRD/TRD + ADR.** (출력 관리 단위 = View 독립, 클라 공유버퍼+per-view 인덱스.) 박막화·코더 분리물은 이 결정 **뒤로 보류.**

## ★이번 세션 핵심 전환 (가장 중요)★
- **두 축을 섞고 있었다(어긋남의 근원):** ① "구독 권위 = 누구냐"(WebView vs src-tauri) = 갈림길 ②(a), 이미 결정됨. ② "구독·추적·재구독의 **단위** = 에이전트냐 View냐". 직전까지 ①만 정리하고 ②(에이전트당 단일 구독+fan-out+min 합산 구조)는 그대로 둔 채 박막화를 그 위에서 돌려, 계속 "창이 합쳐지는" 결함·프레임이 재생산됐다.
- **사용자 원칙(정본 = study-note 20260629·20260630):** 출력을 에이전트당 단일 seq로 묶지 말고 **View(화면) 단위 독립**으로. 업계도 "콘텐츠는 중앙 단일 보관 + 진행도는 뷰별 독립 인덱스, 전송 시퀀스 ≠ 소비 진행도 분리"가 보편(확실).
- **사용자 설계 안(확정 방향):** 클라(중계 허브)가 에이전트당 **공유 버퍼 1개**(중첩 시 가장 큰 범위) 보관 + 각 View는 **인덱스만** 독립. = VS Code pty host(중계가 replay 버퍼 단일 소유, 각 renderer는 독립 cursor) + Kafka/Redis(공유 로그 + per-consumer offset) 결합.
- **핵심: "가장 뒤처진 View min"의 역할 전환** — 현재는 *데몬 재구독 지점*(결함 유발) → 새 모델에선 *클라 버퍼 보관 하한(eviction)*. 새 View는 데몬 재요청 없이 **클라 버퍼에서 자기 인덱스로** 채움. 클라 버퍼에 없는 더 과거는 데몬 ReplayBuffer 2-tier fallback.

## 정식 설계 때 풀 결정점 (PRD/TRD/ADR 거리)
1. 클라 버퍼 보관 범위 = 붙은 View 최대 필요분, 초과분은 데몬 ReplayBuffer(2MB) 2-tier fallback. 데몬=장기 원본·전송 무손실 / 클라=뷰 채우기 캐시 역할 분담.
2. 인덱스 단위(seq) ↔ 버퍼 내 위치 매핑.
3. 각 View xterm scrollback(렌더 결과) ↔ 클라 버퍼(raw bytes) 관계·중복.
4. 프로토콜 영향: 데몬 구독은 여전히 에이전트당 1개로 충분한가(클라가 fan-out·버퍼·인덱스 다 가지면 데몬은 단일 무손실 스트림만). viewer 식별자 데몬에 불필요할 가능성 높음 → 변경 최소.
