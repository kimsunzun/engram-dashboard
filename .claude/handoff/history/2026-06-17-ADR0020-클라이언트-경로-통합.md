## 2. 통합 아키텍처 (핵심 멘탈모델, 재론 금지)
```
프론트: components/store → AgentClient(인터페이스) → ProtocolClient(구현 1벌)
         ProtocolClient = request_id pending·seq high-water dedup·epoch 가드·resubscribe resume·on* 라우팅 (carrier-무관)
              └ Transport: InProcTransport(invoke agent_command + agent_connect Channel) / WsTransport(WS+재연결)
백엔드: 어댑터(carrier별) → ConnectionCore.dispatch(transport-중립) → AgentManager(공유, tauri import 0)
         WS 어댑터(ws.rs handle_connection): handshake/auth/frame codec/keepalive + WsOutboundSink
         Tauri 어댑터(embedded_carrier.rs): agent_command→inbound mpsc→단일 command loop→dispatch, 단일 writer mpsc→Channel
```
- **핵심: carrier만 다르고 프로토콜+dispatch는 공유.** 새 기능 = AgentCommand variant 1 + dispatch arm 1 + (transport 무관) client 메서드 1. **양쪽 2벌 배선 소멸.**
- 미래 carrier(gRPC/모바일) = 어댑터 1개 추가(OutboundSink 구현 + AgentCommand 스트림 공급).

## 3. ★검증에서 잡은 핵심 결함 2건 (재발 방지)★
- **embedded invoke racing(GPT consult 단독 적출):** Tauri command는 각 invoke가 **독립 async task**라 병렬 도착 시 명령 순서 비결정(WS는 단일 read_task라 자연 직렬). → embedded도 **inbound mpsc + 단일 command loop**로 직렬화(`embedded_carrier.rs`). 우회 분기 금지.
- **R1 FIFO — Tauri Channel은 큐 없음(reviewer 적출):** `Channel.send`는 호출 스레드 동기 실행 + payload>8192B는 fetch 큐로 역전 → command loop(replay)·pump(live emit)가 순서경쟁. → **단일 writer mpsc 큐**(control+output 합류, 전용 task가 순서대로 channel.send 1곳)로 WS conn_tx와 동형화. R1(Ack→replay→ReplayComplete↔live) 자료구조 보장.
- **embedded output epoch 전멸(reviewer 적출):** output이 epoch 0 고정(PtyEvent에 epoch 없음)인데 SubscribeAck는 세션 실제 epoch(resume-fallback 시 ≥1) → epoch 가드가 전부 drop. **PtyEvent에 epoch 추가**해 core OutputFrame.epoch→PtyEvent→TauriOutbound→InProc 전파(WS binary frame epoch와 동형). 옛 EmbeddedClient는 epoch 가드가 없어 멀쩡했던 것 — 프로토콜로 올리며 새 회귀였음.

## 4. 새 파일/심볼 (코드 지도)
