## 미결 설계 결정

- **④-b resubscribe 이중화 (핵심):** 재연결 시 Rust router resubscribe(after_seq=`min_render_seq`)와 ProtocolClient.resubscribeAll(after_seq=`lastDeliveredSeq`)이 **이중 Subscribe** → replay 범위 어긋남 가능(실측 미확인). 출력 재구독 단일 권위를 **Rust로** 둘지 **프론트로** 둘지 결정해야 함. 충돌: Rust 단일로 가려면 carrier가 connected 전이를 ProtocolClient에 안 올려야 하는데, 그러면 Fix-C ①("Rust emit이 재구독 트리거")과 깨짐. **T7d 실측 후 결정.**
- **app:None:** step-log 박제됨. **정식 ADR 채번 필요(메인 몫).** 근본수정(run_connection을 `Option<AppHandle>`로)은 큰 변경이라 미룸.
- **안② 채택 vs TRD §11 안① 괴리:** TauriTransport를 "계약 동형"(안②)으로 구현 — TRD §11 코드안(안①, 계약 비동형)과 다름. **ADR + TRD 문서 갱신 필요.**

## 실패한 접근 / do-not

- **ProtocolClient에 transport 종류 분기 금지** — reply/output은 onMessage로만(WsTransport 동형). ④-b 해결도 이 안에서.
- **reply hang을 데몬 enqueue 실패 승격으로 풀지 말 것** — ADR-0020 R6(behavior-preserving) 위반·정상 연결도 큐 포화로 끊길 위험. → 클라측 30s 타임아웃 채택.
- **doConnect에서 임의 `setState('connected')` 금지** — Rust `daemon-connection-state` emit이 상태 단일 진실원. (Fix-C에서 제거함 — 되살리지 말 것.)
- **TRD §11의 TauriTransport 코드안(안①) 따르지 말 것** — 계약 비동형. 안②가 채택본.
