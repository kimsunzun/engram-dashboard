## do-not (다시 꺼내지 말 것 — 리뷰가 깐 dead-end)
- ❌ **Layout/View를 protocol `Command` enum에 넣기 + 데몬 dispatch가 ViewManager 호출.** ADR-0035("데몬은 View 모름")·0036·0004 정면 위반. 증거: `src-tauri/src/commands/layout.rs:1-2`("§5 LLM 제어 표면(ADR-0035). 사람 클릭 `window.__engramLayout`·LLM 동일 단일 control surface"). 레이아웃은 **src-tauri invoke + emit** 경로지 데몬 프로토콜 안 탐.
- ❌ "레이아웃이 데몬 너머라 LLM 못 닿음" 진단 — 틀림(이미 ADR-0035로 해결). 진짜 갭 = FE가 구 store(`__engramLayout.dispatch`→slotStore)에서 신 src-tauri invoke로 미전환 + LLM 핸들이 구 dispatch 가리킴 → **S14/dashboard2 도메인.**

## 리뷰가 수렴한 "맞는 모양" (재개 시 출발점)
- 분할선 = **crate 경계**: 데몬 protocol crate = Agent / src-tauri = Layout·View. (worktree 머지충돌도 자연 해소.) View vs Layout 세부 분할은 S14 끝까지 **고정 보류**(Codex).
- 결과는 **별 채널 X** — 기존 한 enum의 `request_id 유무` 분기를 **additive 확장**(version bump + 호환 단계, flag-day 금지). 타입드 reply 유지(제네릭 exec는 약함 — SpawnByCwd 결과가 AgentList로 resolve되면 안 됨). **Resize/Subscribe = request_id 없는 제3부류**(fire-forget/stream, SubscribeAck FIFO 불변식 ADR-0020).
- 메인 권고 = (b) "내 영역 **Agent 도메인만** 정리, Layout은 S14에 위임" — 사용자 최종확정 보류("이벤트버스 완료 후 보자"로 사실상 (b)/(c) 채택).
