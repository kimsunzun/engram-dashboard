## ★ 확정 아키텍처 (3층 — 사용자 결정) ★
```
① 채널 (교체 가능 seam):
   - M3 마커 = 기본. in-band 출력 텍스트에 단일라인 센티넬. 전 모드 커버(특히 터미널 모드).
     하드닝(제어문자/nonce/최종 assistant-text만 파싱)으로 히트율↑.
   - strict-schema tool = 교체 슬롯("다른 방향"). json 모드에서 구조 히트율 최상(constrained decoding).
     조건부·미검(스파이크 대상).
② 검증 레이어 (채널-독립, 데몬 소유): 파싱 → 검증(수신자 실존·스키마·상한) → 거부/로그.
③ 배달 (채널-독립, 데몬 소유): stdin 주입(engram 이미 write_stdin 보유, manager.rs:638).
```

- **핵심 통찰(반복 수렴):** 마커냐 툴이냐는 **"히트율(구조 well-formed + 오발화 방지)"만** 가르고, **"의미 정확성(수신자 맞나·내용 옳나)"은 어느 채널이든 ②검증 레이어가 최종 방어.** → 검증을 채널 밖 공통 레이어에 두는 게 핵심 설계. 사용자의 "양식 잘못은 어느 방법이든 다 마찬가지" 통찰이 이걸 확정.

## tool_use vs 마커 — 왜 툴이 히트율 높나 (긴 Q&A 결론, 기제 이해)
- **파이프라인 경계:** ①LLM(신경망, Anthropic 서버) = 토큰만(텍스트·툴이름·인자값). → ②Anthropic API 서버 = raw 토큰 파싱→**공개 JSON 형상화**(`{type:"tool_use",id:"toolu_",input:{}}`)·id 발급·type 태깅·strict 제약. **JSON은 여기서 탄생, 네트워크를 JSON으로 건넘.** → ③claude.exe(내 PC, **모델 아님·원격 API 클라이언트**) = 받은 JSON을 터미널이면 텍스트 렌더, stream-json이면 겉봉투(`type:"assistant"/"result"`+session_id) 씌워 stdout. → ④engram 데몬이 stdout 파싱.
- **툴 우위 2축:** (a) LLM이 *툴콜 형식*으로 커밋 → 서버가 `type:"tool_use"` 태깅 → 데몬은 **타입만 봄**(설명·예시 텍스트와 안 섞임 = **오발화 0**). (b) **strict면 args JSON에 constrained decoding** — 매 토큰 스텝에서 스키마 위반 토큰을 확률 0으로 마스킹(logit −∞) → 무효 못 냄 = **구조 하드 보증**(단 LLM 자체가 아니라 추론 엔진이 걸어줌).
- **마커 = LLM이 text 형식으로 낸 것** → `type:"text"` 태깅 → 데몬이 텍스트 안에서 스크랩(설명 예시 섞여 오발화 위험) + strict 없음(soft).
- **단 조건:** (a) 데몬이 tool_use 보려면 **stream-json 모드 필요 — 터미널(TUI) 모드선 못 봄**(텍스트로 렌더) → 그래서 마커가 전 모드 답 = M3 존재 이유. (b) strict 실제 적용 **미검**. (c) strict는 구조만 강제, **의미(수신자 실존)·max_tokens 잘림·refusal**은 여전히 검증 필요.
- **stream-json 실체(다음 세션 검증):** tool_use 블록은 `input:{}` 빈 채로 열리고 args가 `input_json_delta.partial_json` 문자열 조각으로 스트리밍 → 누적 후 파싱. text는 `text_delta`. stop_reason: tool_use/end_turn/max_tokens/refusal.

## Codex 적대리뷰 판정 = BLOCK (M3 "성급 폐기" 교정 — cross-family가 값을 함)
- "아무도 마커 안 씀" = 과장/표본 부실(코퍼스·방법 미명시).
- Orca가 per-call CLI spawn 감수 → 속도 반론이 오히려 약화되나 **engram-특이 고빈도 제약은 측정 필요**(비-refute).
- "구조화 tool_use가 false-trigger 우회" = **어휘 오탐만 제거**, 프롬프트 주입/수신자 위조/리플레이는 못 막음(critical).
- "bracketed-paste 어차피 필요" = **거짓**(Agent Teams·Channels는 paste 불필요; Orca PTY 기법이 headless stream-json엔 안 옮겨질 수 있음).
- **진짜 결정 기준 = 배달 시맨틱**(순서·ACK·busy·복구)이지 마커vs툴 문법 아님(completeness).
- **M3의 진짜 강점(과소평가됨) = 발신 프로버넌스가 데몬-소유 stdout 스트림에 자연 결속 — 새 쓰기 권한 불요.** 대안은 이 속성을 명시적으로 대체해야.
- 결론: peer 근거는 재검토 정당화하나 **대체안 선택은 못 함 → M3 폐기 부당, 하드닝 M3가 정당.**

## NEEDS_VERIFICATION (다음 세션 스파이크 — Codex가 넘긴 것 + 우리 추가)
1. **Claude Code stream-json 실제 봉투** — `type:"assistant"/"result"/"system"` 형태 맞나 · tool_use가 content 블록으로 오나 · `result`가 턴완료냐 프로세스완료(-p 원샷)냐 · 지속 프로세스에서 "다음 입력 대기" 경계 있나 · `--include-partial-messages` 필요한가. **스파이크: `claude -p --output-format stream-json --verbose`로 툴콜 하나 찍어 관찰.**
2. **strict/constrained decoding** — Claude Code가 MCP/커스텀 툴을 strict로 API에 넘기나(넘기면 교체슬롯 tool이 구조 하드보증 획득).
3. **제어문자 보존** — CLI가 stdout에 raw 제어문자(RS 0x1E / STX 0x02) 보존하나(마커 하드닝 A안 게이트 — 안 되면 인쇄가능 랜덤토큰 B안).
4. **에이전트 토큰 재현** — 마커에 세션 랜덤토큰 넣으면 에이전트가 설명 중 재현하나(B안 확률적 방어 검증).
5. Agent Teams 메일박스 경로·락·버전(문서 vs 추론) · MCP Channels 정확 상태·플래그·버전 · ✳ idle이 Claude 공식 계약이냐 Orca 휴리스틱이냐 · Orca 소스 permalink · tmux ~70-80% 신뢰도 출처.
6. **Codex web_search** — config.toml 영구설정 + MCP 재시작 후 뜨는지(위 로컬변경).
