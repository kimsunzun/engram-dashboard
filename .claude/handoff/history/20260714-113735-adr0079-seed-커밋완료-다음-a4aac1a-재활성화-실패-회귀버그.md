## ★ 다음 과제 = `a4aac1a` 재활성화 회귀 (root-caused, 미수정) ★
**증상(유저 실측 스크린샷):** 터미널(PTY)·제이슨(StreamJson) 에이전트가 정상 실행 중(green, 콘텐츠 표시)인데, **"다시 실행"(재활성화)하면 둘 다 `실패`(Failed) + 슬롯이 "에이전트 연결 중…" 빈 화면**으로 뒤집힘. seed와 무관, 모드 무관(공통 경로).

**원인(결정적, 코드+git 확정):** 커밋 `a4aac1a`("수동 활성화도 resume 조기종료 시 fresh-fallback", ADR-0077)가 `SpawnProfile` 핸들러를 가드 없는 `activate_profile`로 우회시킴. 체인:
1. 재활성화 → `activate_profile`(`manager.rs:286`) → `resume_with_fresh_fallback`(`:488`) → `spawn_agent(Resume)`.
2. `spawn_agent`의 **이중 spawn 가드**(`:167-178`)가 이미 맵에 있는 산 에이전트를 보고 `Err("agent … already running")` 반환.
3. **결함:** `resume_with_fresh_fallback`(`:489-490`)이 이 **"already running" 가드 Err를 "resume 프로세스 실패"로 오인** → `fallback_fresh`(`:514`) 호출.
4. `fallback_fresh` → `remove_session`(`:584`)이 **멀쩡히 돌던 에이전트를 kill** → epoch++ → `spawn_agent(Fresh)`.
5. 그 fresh 재spawn이 실패/조기종료면 → `RestoreOutcome::Failed`(`:544`) → `activate_profile` Err(`:306`) → **Failed 상태·빈 슬롯**. 성공해도 원래 대화 세션은 파괴되고 빈 fresh로 교체(epoch+1 재구독 → placeholder).
- **결정적 진입**(가드 Err→항상 fallback_fresh→항상 산 에이전트 kill). 최종 Failed 여부만 fresh 생존/실패에 따라 비결정. remove_session vs Fresh 가드 레이스는 **아님**(remove_session이 join_pump까지 블록 후 반환).
- **`a4aac1a` 이전:** `connection_core.rs`가 `spawn_agent` 직접 호출 → 재활성화 = 가드 Err → 무해한 "already running" 토스트, **산 에이전트 그대로.** a4aac1a가 그 무해 Err를 "실패→파괴+fresh-fallback"으로 바꿈.

**핵심 결함 한 줄:** `resume_with_fresh_fallback`(`manager.rs:489`)이 두 종류 Err를 뭉갬 — "resume 프로세스 launch/조기종료 실패"(fresh-fallback 맞음) vs **"already running" 가드 Err**(아무것도 하지 말아야 함).

**수정 방향(미구현 — critical 경로):**
1. **가드 Err는 fresh-fallback 금지** — `spawn_agent` 가드가 generic `SpawnFailed("… already running")`(`:174`) 대신 전용 `PtyError::AlreadyRunning` 반환하게 하고, `resume_with_fresh_fallback`은 그 변형이면 파괴 안 하고 short-circuit(기존 반환/에러). pre-a4aac1a 불변식 복원(국소·안전).
2. **재활성화 UX = 사용자 결정** — 실행 중 에이전트 재활성화 = 포커스/무동작 vs 명시적 재시작. `activate_profile`(`:286`)/`SpawnProfile` 핸들러(`connection_core.rs:973-987`)에서 `sessions.contains_key(id)` 선검사.
- a4aac1a의 테스트(`tests/activation.rs`)는 reserved/early-exit(shell)만 커버 — **already-running 재활성화 케이스 회귀 테스트 추가 필요**(CI 통과해버린 이유).
