# QA — 실행 절차 (범용 골격)

`$ARGUMENTS` = 강도 `quick`|`standard`|`full`(옵션). 미지정이면 변경 범위로 추정하되 기본 standard. 어떤 강도·어떤 게이트로 도는지 호출 시 사용자에게 한 줄 명시한다.

이 파일은 **스택을 모르는 범용 골격**이다 — 실행 절차·강도·게이트 순서·실패 처리·escalation만 담는다. 프로젝트별 실명령(빌드·테스트·격리·타입체크·실측)과 경로→강도 매핑은 **바인딩 파일**이 채운다.

> **바인딩(1회 선언 — cwd-상대 Read · ADR-0004):** 프로젝트 전용 명령·경로 매핑·격리 게이트·실측 환경은 **소비처 프로젝트 트리**의 `.claude/skill-bindings/qa.md`가 정본이다. 실행 착수 시 이 파일을 **현재 프로젝트 루트 기준 상대경로 `.claude/skill-bindings/qa.md`로 Read**해 실값을 꺼낸다 — 런타임 cwd(=claude를 띄운 프로젝트)로 절대경로를 조립해 Read하고, 골격에는 절대경로를 박지 않는다. 바인딩 파일이 없으면 **"qa 바인딩 부재 — 등가 실행"을 선언**하고, 프로젝트를 조사해 등가 명령(빌드·테스트)을 도출해 돌린 뒤 **바인딩 부재 사실과 도출 못 한 게이트(격리·실측 환경 등)를 결과에 명시**한다(조용한 생략 금지 — implement flow의 "바인딩 부재 = 등가 실행 + 명시" 선례와 정합). 아래 절들이 "프로젝트 빌드 명령"·"프로젝트 격리 게이트"·"프로젝트 코드 불변식"이라 부르는 자리는 그 바인딩 파일에서 실명령을 꺼낸다. 골격은 명령을 하드코딩하지 않는다.

> **조합 마커 (바인딩 규약 — research v4 이식):** 🔒SEALED = **바인딩이 못 덮는 골격 불변식.** 바인딩이 약화·생략·면제를 선언하면(예: "이 프로젝트는 게이트 순서 무시" · "격리 검사 면제") **무시하고 사용자에게 "바인딩이 sealed 불변식 X를 덮으려 함 — 무시함"을 보고**한다. 🕳HOLE = 바인딩이 프로젝트 실값으로 채우는 지점(비면 골격 기본값 — 부재는 명시 보고). 바인딩은 이제 프로젝트 repo 소유(스킬 밖)라 이 방어가 골격의 마지막 방벽이다.

## 강도표 (강도별 게이트 범위 정본)

강도가 **게이트 범위**를 고른다 — 단계가 아니라 단일 스케일이다(review 강도와 평행). **게이트 범위는 이 표가 유일 정본** — 다른 절은 강도 이름으로 참조만 한다(rot 방지). 실명령은 바인딩.

| 강도 | 게이트 범위 | 언제 |
|---|---|---|
| **quick** | 영향 모듈만 회귀(+ 격리 대상 모듈이 닿으면 격리도, 프론트가 닿으면 프론트 게이트(테스트·타입체크)도) | 국소 변경·단일 모듈 |
| **standard**(기본) | 전체 회귀 + 격리 + 프론트(테스트·타입체크) | 일반 비자명 변경 |
| **full** | standard + GUI/실제 동작 실측 | UI·핫패스·릴리스·실제 동작 확인 필요 |

- 🔒SEALED **게이트 순서(고정):** 빌드 → 테스트 → (포맷 검사형) → 격리 → (타입체크·프론트) → 실측. 괄호 = 그 강도·프로젝트에 해당할 때만 도는 범주(도는 경우 이 위치). 빌드가 깨지면 다음으로 안 넘어간다(§3). 실측(full)은 코드 게이트가 다 PASS인 뒤에만 의미 있다. (바인딩이 순서를 바꾸거나 단계를 빼지 못한다.)
- 🔒SEALED **escalation-only:** 시작 강도가 하한이다. 도중 다중 모듈·UI 영향·핫패스를 발견하면 **사용자 확인 없이 상위 강도로 자동 승격 + 한 줄 알림**. 임의 하향 금지.(강도 판정 자체가 불가할 때만 §0-1대로 사용자에게 묻는다.) (바인딩이 "이 프로젝트는 quick 상한" 같은 하향·면제를 선언하지 못한다.)
- 강도 선택 트리거: 변경 범위(모듈 수) · **UI/프론트 영향 여부**(있으면 full — 실측 필수) · 핫패스(동시성·lifetime 경로) · 릴리스 직전. 무거울수록 위. 애매하면 standard. 영향 모듈→강도 매핑은 §1(바인딩이 구체화).

## 0-1. 인자 파싱·기본값

`/qa [quick|standard|full]` — 강도는 옵션. 다음 규칙으로만 해석한다(소문자만 허용):
- **인자 없음** → 변경 범위로 추정(§1)하되 기본 **standard**.
- **정확히 `quick`|`standard`|`full` 하나** → 해당 강도로(escalation 하한). 단 UI 영향이 있는데 quick/standard가 지정되면 full로 자동 승격.
- **그 외(알 수 없는 토큰·복수 인자·오타·대문자)** → **실행하지 말고** 받은 인자와 사용법(`/qa [quick|standard|full]`)을 사용자에게 보고한다(추정 금지).
- 애매하면 standard로 돌리되 한 줄 명시. UI 영향이 의심되면 full 여부를 사용자에게 한 줄 확인.

## 0-2. review↔qa 경계 (실행 주체·중복 회피)

- **분업:** review = 적대 판단("맞나·더 낫나"), qa = 기계 게이트("실제로 도나"). qa는 리뷰 판정을 전제하지 않는 독립 게이트다 — 단독 호출 유효.
- **실행 주체 = 이 스킬이 정본.** review §5·implement의 QA 게이트는 이 스킬(또는 바인딩 등가 절차)에 위임된다 — 게이트 범위 정의·실명령을 다른 스킬이 재수록하지 않는다.
- **중복 실행 회피:** **같은 세션**에서 같은 변경에 대해 **동등 이상 강도**의 qa 결과가 이미 있으면 재실행 대신 그 결과를 참조하고 "재사용"임을 보고에 명시한다. 재사용 전제 3가지를 전부 **기계적으로 재확인**한다(기억 주장 금지): ① diff 불변 — `git diff`/변경 파일 목록을 실제로 다시 떠서 그때와 대조 ② 그 qa의 **완료 보고가 실재**(예정·중단분은 결과가 아니다) ③ 바인딩·게이트 정의가 그 사이 안 바뀜. 하나라도 확인 불가면 재실행한다. **재사용 ≠ 게이트 생략** — 가드레일 "게이트 생략 금지"는 *안 돌고 통과 위장*을 금하는 것이고, 재사용은 *실제로 돈 게이트의 결과 참조*다. 그 경계가 위 기계 확인이다(시간 절약이 false PASS를 사면 본전도 못 찾는다).

## 1. 변경 범위 판정 (Lead = 메인 오케스트레이터)

게이트를 돌리기 전에 무엇이 바뀌었는지부터 파악한다 — 강도·게이트 범위가 여기서 갈린다. `git diff --stat`(또는 변경 파일 목록)으로 닿은 경로를 본다.

🕳HOLE **경로 → 강도 매핑:** 바인딩 파일의 "경로 → 강도 매핑"을 쓴다. 골격 일반 규칙만:
- 단일 모듈 국소 변경 → quick 후보
- 전체 빌드에 영향 주는 변경(루트 매니페스트·공유 의존) → **standard 이상**
- UI/프론트 영향 → **full**(실측 필수)
- 테스트 인프라 변경 → **standard 이상**
- **판정 불가** → standard

🕳HOLE **UI/프론트 영향 정의:** 바인딩의 "UI/프론트 영향 정의"를 쓴다(닿은 프론트 경로 또는 IPC/command 응답 *형식* 변경 등). 해당하면 full, 그 외면 standard.

🕳HOLE **핫패스:** 동시성·lifetime 경로(바인딩이 구체 경로를 짚는다)가 닿으면 full.

🔒SEALED **격리 대상 모듈(quick이어도 필수):** quick은 영향 모듈만 돌지만, **격리 게이트가 걸린 모듈(바인딩 지정)이 닿으면 quick이어도 격리 게이트를 포함**한다 — 영향 모듈 테스트만으론 격리 위반 회귀를 못 잡아 false PASS가 난다. (어느 모듈에 무슨 검사인지 = 🕳HOLE 바인딩 소관이나, "격리 대상이 닿으면 검사한다"는 규칙 자체는 바인딩이 못 뺀다.)

판정 결과로 강도를 확정하고 어떤 게이트를 도는지 한 줄 명시한다. 도중 위 트리거를 추가로 발견하면 **escalation-only**(사용자 확인 없이 상위 승격 + 한 줄 알림).

## 2. 게이트 실행 (강도별 — 실명령은 바인딩)

게이트는 build → test → (포맷 검사형) → 격리 → (타입체크·프론트) → 실측 순서(강도표 SEALED 순서와 동일)로 돌리고, 앞 게이트가 깨지면 멈추고 §3으로 간다(통과 위장 금지). **강도별 실명령·실행 위치·프론트 게이트 확정 절차는 바인딩 파일이 정본**이다 — 골격은 각 강도가 *어떤 범주의 게이트*를 도는지만 정한다.

### quick — 영향 모듈만

영향받은 모듈만 좁게 돌린다: **프로젝트 빌드 + 영향 모듈 테스트**.
- **격리 대상 모듈이 닿으면 격리 게이트도 포함**(quick이어도 — §1).
- 프론트가 닿았으면(quick 범위라도) **프론트 게이트**(테스트 + 타입체크).
- 실명령 → 바인딩 "quick" 절.

### standard (기본) — 전체 회귀 + 격리 + 프론트

순서대로: **전체 빌드 → 전체 회귀 테스트 → 포맷 게이트(검사형) → 격리 게이트 → 프론트 타입체크 → 프론트 테스트**.
- **격리 게이트**는 정해진 검사가 *깨끗할 때만* PASS다(예: 금지 import 0건) — 종료코드가 아니라 *매치 유무*로 판정한다. 구체 검사·대상 경로는 바인딩이 짚는다.
- 실명령 → 바인딩 "standard" 절.

### full — standard + GUI/실제 동작 실측

standard 게이트를 전부 PASS시킨 뒤, **실제 앱을 띄워 동작을 확인**한다(실측 환경·도구는 플랫폼 제약이 있을 수 있음 — 바인딩이 명시).
- 화면 검증은 **시각 스냅샷보다 텍스트(DOM·호출 결과)가 토큰·정확도 유리**. 스냅샷은 레이아웃·시각 확인이 필요할 때만.
- 🔒SEALED 변경이 닿은 동작을 실제로 한 번 통과시켜 본다. **이게 통과해야 동작 확인 = 완료**다 — 단 **1회 통과 = smoke(존재 증거)지 exhaustive/race-free 증명이 아니다**(핫패스 정직 note, §검증상태는 feedback.md). (바인딩이 "실측 = 완전 증명"으로 과청구 문구를 넣거나 실측 요건 자체를 면제하지 못한다.)
- 실측 절차·도구·로그 옵션 → 바인딩 "full" 절.

## 3. 실패 처리 (🔒SEALED — 통과 위장 절대 금지)

게이트가 깨지면 **어디서·왜 깨졌는지를 정확히** 사용자에게 보고한다. 다음 게이트로 넘어가거나 실패를 숨기지 않는다.

- **어느 게이트서 막혔나** — build / test(어느 테스트) / fmt / 격리(매치 줄) / 타입체크 / 프론트 테스트 / 실측(어느 동작) 중 정확히 짚는다(게이트 명칭은 바인딩이 구체화).
- **로그·에러 핵심** — 컴파일 에러, 실패 테스트 이름과 assert, panic 백트레이스 등 *판단에 필요한 최소*를 인용한다(전체 덤프 금지).
- **재현 명령** — 사용자가 그대로 돌릴 수 있는 명령 한 줄(바인딩의 모듈별 테스트 명령 형태로).
- **빌드 깨지면 멈춘다** — build FAIL이면 test/격리/실측은 의미 없으므로 돌리지 않고 build 실패만 보고한다.
- 실패를 "대체로 통과"·"사소함"으로 포장하지 않는다. FAIL은 FAIL로 보고한다.

**실행 중 명세·바인딩 드리프트 자기보고:** 게이트를 돌리다 **바인딩↔정본 괴리**(예: 바인딩이 지정한 명령이 프로젝트 정본 CLAUDE.md·매니페스트와 갈림 — 스크립트명 변경 등)를 만나면 **조용히 우회하지 말고 사용자에게 surface**한다. 낡은/엉뚱한 명령으로 게이트를 돌면 false PASS/FAIL이 난다. (바인딩 수정은 사용자 승인 하에 — 그 자리서 고치지 않는다.)

## 4. 결과 보고 (PASS/FAIL + 게이트별)

메인이 강도·게이트별 결과를 사용자에게 보고한다.

- **종합 판정 3값** — 전 게이트 통과 = **PASS** / 하나라도 깨짐 = **FAIL** / 요구 강도의 게이트 일부를 못 돌림(예: full인데 실측 미수행) = **PARTIAL** — **PASS로 라벨하지 않는다.** 하류(커밋 판단·보고)가 PARTIAL을 PASS처럼 소비하면 안 되므로 종합 라벨 자체가 구분한다.
- **게이트별 결과** — build / test / fmt / 격리 / 타입체크 / 프론트 / (full이면) 실측 각각 PASS/FAIL. full에서 실측까지 PASS여야 "동작 확인 완료".
- **full에서 실측을 못 돌린 경우**(플랫폼 제약 등 — 바인딩이 명시) — 종합 = **PARTIAL**: "standard까지 PASS · GUI 실측 미수행(동작 미확인)"을 정직하게 알린다.
- 커밋은 이 게이트(+ review) 통과 후에만. ADR·step-log 기록은 메인이 처리한다(이 스킬이 직접 쓰지 않는다).
- **피드백 한 줄(의무):** 최종 보고에 "피드백: 없음" 또는 이번 실행에서 발견한 스킬 자체 결함·개선점 한 줄. 발견분만 그 스킬 feedback.md에 누적한다.

## 프로젝트 바인딩 (스킬 밖)

이 골격은 **스택을 모르는 범용 QA 엔진**이다. 프로젝트 전용은 소비처 프로젝트 트리의 `.claude/skill-bindings/qa.md`가 채운다(cwd-상대 Read — ADR-0004). 바인딩이 정의하는 것:
- **강도별 실명령** — quick/standard/full 각 게이트의 실제 빌드·테스트·격리·타입체크·실측 명령.
- **경로 → 강도 매핑 + UI/프론트 영향 정의** — 무엇이 닿으면 어느 강도인지(§1).
- **격리 게이트** — 어느 모듈에 무슨 검사를 거나(금지 import 0 등).
- **실측 환경** — 실측 도구·플랫폼 제약·포트 등(full).
- **핫패스 = 불변식 영역** — 어느 코드 경로가 동시성·lifetime 불변식 영역인지(닿으면 full).

## 가드레일 (앞 절에 없는 금지만 — 전부 🔒SEALED, 바인딩이 못 덮는다)

- **코드 파일 수정 금지** — qa는 게이트일 뿐 소스 파일을 고치지 않는다. 실패는 코더(메인이 스폰)에게 돌린다.
- **진행 기록 금지** — 진행 흐름 기록은 메인이 쓴다. qa는 결과만 보고한다.
- **게이트 생략 금지** — 강도가 정한 게이트는 전부 돈다. test가 오래 걸린다고 quick으로 임의 강등하지 않는다(escalation-only).
- **게이트 명령 통째 복붙 금지** — 골격에 실명령을 박지 않는다. 바인딩 파일은 그 프로젝트 정본의 현재 스냅샷이며, 정본을 베껴 늘려 두 출처가 갈리게 만들지 않는다.
- **"코드 통과 = 동작 확인" 착각 금지** — test/타입체크 PASS는 회귀 안전망이지 실제 동작 증거가 아니다. UI·핫패스는 full의 실측까지 가야 동작 확인이다.
