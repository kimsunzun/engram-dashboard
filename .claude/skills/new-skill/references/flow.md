# New Skill — 실행 절차

`$ARGUMENTS` = `["<스킬명>"] [--global|--project]`. 둘 다 옵션.

## 0. 인자 파싱 + 스킬명 중복 검사

- **스킬명** — 있으면 그대로. 없으면 §1 인터뷰에서 도출.
- **위치 플래그** — `--global/--project` 있으면 기록. 없으면 §1에서 추정 근거를 모아 §2 브리핑에서 사용자 확인.
- **중복 검사** — 스킬명이 정해지면 글로벌(`~/.claude/skills/`)·프로젝트(`.claude/skills/`) 양쪽에 동명 폴더가 있는지 확인한다. 있으면 즉시 사용자에게 알리고("기존 스킬 `<이름>` 존재 — 덮어쓸지 확인") 확인 전 진행 중단.
- 호출 시 "어느 스킬을 만드는지 + 중복 검사 결과"를 사용자에게 한 줄 명시한다.

## 1. 발견 인터뷰

아래 질문을 **한 번에 묶어** 제시한다(왕복 최소화):

```
1. 이 스킬이 해결하는 반복 작업은? (한두 줄)
2. 스킬 없이 하면 어떤 문제가 생기나? (복붙 오류·컨텍스트 오염·절차 누락 등)
3. 트리거 조건은? (사용자가 /스킬명을 치는 상황 2~3가지)
4. 인자가 있나? (강도·오퍼레이션·대상 등 — 없으면 없다고)
5. 산출/결과물은? (파일·보고·에이전트 스폰·명령 실행 등)
6. 이 프로젝트 전용인가 범용인가? (특정 스택·경로·ADR 참조 여부)
```

답변 후:

**스킬화 가치 점검** — SKILL.md §스킬화 가치 기준 4개 중 2개 이하 충족이면:
> "기준 N개 충족 — 스킬보다 CLAUDE.md 메모 또는 scripts/ 유틸이 적합할 수 있습니다. 계속 진행할까요?"

사용자가 진행 원하면 계속. 중단이면 여기서 종료.

**위치 추정** (플래그 미지정 시) — 질문 6 답변으로:
- "특정 스택·경로·ADR 참조 있음" → 프로젝트 스킬 추정
- "범용 흐름" → 글로벌 스킬 추정

추정 결과를 §2 브리핑에 포함해 사용자 확인을 받는다(자동 확정 금지).

## 2. 인터페이스 설계 + 브리핑 (1회 승인)

인터뷰 답변으로 아래 항목을 채운다:

```
스킬명:         <name>
위치(추정):     글로벌 | 프로젝트(<root>/.claude/skills/) — 사용자 확인 필요
1차 축:         강도 | 오퍼레이션 | 단계 | 없음
인자 파싱:      <파싱·기본값 규칙>
트리거 문장:    /스킬명 [인자...] — <한 줄>
절차 단계:      §1 ~ §N (단계명 + 한 줄 요약)
산출:           <파일·보고·에이전트 결과 등>
생성 파일:      <절대 경로 목록>
bindings 필요:  있음(<이유>) | 없음
LLM 제어 필요:  있음(<어떤 기능이 제어 표면 필요한지>) | 없음
```

**1차 축 선택 기준:**
- **강도** — 같은 흐름을 깊이로 조절 (research/review/qa 패턴)
- **오퍼레이션** — 뚜렷이 다른 행동 유형 (adr: new/supersede/lint)
- **단계** — 파이프라인 위치가 동작을 결정 (review: prd/trd/code/doc)
- **없음** — 단일 흐름, 분기 없음 (continue 패턴)

이 항목들을 **브리핑 형식으로 사용자에게 제시**하고 한 번에 확인받는다:
- 위치 확정
- 생성 파일 목록 확인
- LLM 제어 표면 필요 여부 확인

확인 전 파일 생성 금지. 이 단계가 §3 생성 전 마지막 관문이다.

## 3. 파일 생성

### 포맷 참조 (생성 전 필수 Read)

파일 작성 전 아래 두 파일을 Read한다 — 포맷·섹션 순서·금지 패턴의 단일 출처다. 기억에서 꺼내거나 발명하지 않는다.

§0에서 확인한 스킬 루트 경로 기준 절대 경로로 Read한다 (예: 글로벌이면 `C:\Users\<user>\.claude\skills\new-skill\references\formats\`, 프로젝트이면 `<root>\.claude\skills\new-skill\references\formats\`).

- `references/formats/SKILL.template.md` — SKILL.md 표준 포맷
- `references/formats/flow.template.md` — flow.md 표준 포맷

### 파일 구조

**기본 (bindings 불필요):**
```
skills/<스킬명>/
  SKILL.md
  references/
    flow.md
  feedback.md
```

**bindings 포함:**
```
skills/<스킬명>/
  SKILL.md
  references/
    flow.md
    bindings/
      <project>.md
  feedback.md
```

**학습용 (복잡한 추론 누적 필요 시 — 신규 스킬이 cross-family 교차검증·다단계 판단 포함):**
```
skills/<스킬명>/
  SKILL.md
  references/
    flow.md
  study-notes/    ← 한시적. 사용자 "이해됐다" 명시 시 제거.
  feedback.md
```

study-notes는 단순 선형 흐름 스킬엔 기본 생성하지 않는다 — 빈 디렉토리 쓰레기 방지.

### SKILL.md 작성 규칙

**필수 섹션:**
1. Frontmatter (`---`): `name`, `description` (한 줄, 트리거 포함 — 상주 로드되므로 간결하게)
2. `# <스킬명>` + **flow.md Read 강제 문구(H1 직후, 첫 `##` 이전 — lint 위치 검사 대상)** + 한 단락 목적 요약
3. `## 핵심 설계 (불변)` — 핵심 불변 설계 결정 2~4개
4. `## 트리거` — `/스킬명 [인자]` (Read 경고는 항목 2 위치에 이미 있으므로 여기 중복 금지)
5. `## 자기개선 피드백` — _shared 포인터 + **검증 상태는 feedback.md에 기록**

**선택 섹션:**
- `## <1차 축> 표` — 강도/오퍼레이션/단계가 있을 때만
- `## 프로젝트 바인딩` — bindings가 있거나 글로벌 기본값 명시 필요 시

**금지:**
- description에 절차 상세 — 상주 로드되므로 토큰 낭비
- SKILL.md에 flow.md 내용 복붙 — rot 유발
- **SKILL.md에 `## ⚠️ 검증 상태` 절 — 검증 상태는 feedback.md의 "검증 상태" 항목으로**(방침 C)
- 빈 usage-log.md 사전 생성 — 고아 파일. feedback.md는 검증 상태 기록으로 생성 대상이다

### flow.md 작성 규칙

**필수 섹션:**
1. `## 0. 인자 파싱·기본값`
2. 단계별 실행 절차 `## 1.` ~ `## N.`
3. `## 가드레일` — 앞 절에 없는 금지만 (중복 금지)

**원칙:**
- 실 명령(cargo test, npm test 등)은 bindings에 — flow.md에 하드코딩하면 rot
- 선택지 제시 시 AskUserQuestion 또는 브리핑 — 임의 채택 금지
- SKILL.md의 flow.md Read 경고와 쌍을 이룬다 — flow.md가 정본이어야 경고가 의미 있다

### 비자명성 판단 (인라인 vs 서브에이전트)

스킬 파일 작성은 에이전트 동작을 정의하는 명세라 기본적으로 비자명하다. 단, 현재 세션에 설계 컨텍스트가 완전히 있고 내용이 §2 브리핑에서 이미 확정됐으면 메인이 직접 작성한다 — 서브에이전트로 넘기면 컨텍스트 손실이 오히려 품질을 낮춘다.

## 4. 후속 안내

생성 완료 후:
1. **생성 파일 목록** (절대 경로)
2. **바로 다음 권장:**
   - `/review trd` — 비자명 스킬이면 강력 권장
   - LLM 제어 표면이 필요하다고 판단됐으면 해당 구현 안내
3. **_shared 확인** — 프로젝트 스킬이면 `_shared/self-improvement-feedback.md` 존재 여부 확인

## 가드레일

- **인터뷰 생략 금지** — 스킬명이 인자로 왔어도 §1은 돈다. 인터뷰 없으면 flow.md가 껍데기.
- **위치 자동 확정 금지** — §2 브리핑에서 사용자 확인 전에 위치를 결정하지 않는다.
- **브리핑 전 파일 생성 금지** — §2 확인 전 파일을 건드리지 않는다.
- **formats/ 미Read 생성 금지** — §3 포맷 참조 두 파일을 Read하기 전에 SKILL.md·flow.md를 생성하지 않는다. "기억에서 꺼낸" 포맷은 생태계 불일치를 만든다.
- **기존 패턴 임의 교체 금지** — flow.md Read 경고·검증 상태 기록(feedback.md)은 생태계 일관성 구성요소. 생략하려면 사용자 확인.
- **description 비만 금지** — 절차 상세는 flow.md. description은 트리거 포함 한 줄.
- **빈 파일 사전 생성 금지** — usage-log.md는 실제로 쓸 내용이 생겼을 때 만든다. feedback.md는 예외 — 검증 상태(방침 C)를 담아 처음부터 생성한다(lint의 검증 상태 검사 대상).
