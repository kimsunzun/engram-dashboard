# Implement

비자명 코드 변경을 **코더 서브에이전트 스폰 → `/review code` 게이트 → `/qa` 게이트 → 커밋 준비**로 한 호출에 오케스트레이션한다. 이 스킬의 존재 이유는 **메인이 직접 구현하는 것을 구조로 차단**하는 것이다 — 위임 우선(메인 토큰은 판단에), 생산자≠리뷰어(짠 사람이 자기 코드를 통과시키지 않게), 게이트 순서·모델 배정·루프 상한 표준화. 선례 근거 = Anthropic 공식 Orchestrator-Subagent 패턴(순서가 알려진 파이프라인의 기본형).

## 핵심 설계 (불변)

- **메인은 오케스트레이터 — 구현을 직접 짜지 않는다.** 라우팅·지시서·취합·보고만 메인 토큰이 쓴다. 실제 편집은 fresh 코더 서브에이전트가 한다. 이게 권장이 아니라 이 스킬의 정체성이다 — 자율 모드("진행 쭉해")에서도 생략 금지.
- **생산자 ≠ 리뷰어 (fresh 분리).** 코더가 짠 산출물은 짠 코더가 아니라 `/review code`(다른 family 적대 리뷰)가 때리고, `/qa`(기계적 게이트)가 실측한다. 코더는 자기 코드를 통과시키지 못한다.
- **게이트 순서 불변 — review → qa.** review("맞나", 적대 판단)가 먼저, qa("실제로 도나", 기계 게이트)가 후행. 역전·생략 금지 — qa는 review의 후행 짝이다(두 게이트 스킬의 계약).
- **강도 = 라우팅(1축).** simple/standard/critical이 코더 슬롯(단순/복잡)과 두 게이트 강도를 한 번에 정한다. 정량(어느 게이트 강도로 넘기나)은 `references/flow.md` 강도표가 정본 — 아래 표는 "언제"만.
- **루프 상한 = 무한 리뷰 방어.** 코더 재수정(review FIX·qa 실패 합산) 상한 2회. 초과 시 임의로 계속하지 않고 사용자 에스컬레이션. 정량·리셋 정책은 `references/flow.md §3`.
- **커밋 준비까지만 — 커밋은 호출자 몫.** 게이트 통과 후 커밋 메시지 초안 + 스테이징 대상만 메인에 반환한다. 커밋 실행·기록(ADR/step-log)은 호출자(메인)와 프로젝트 관례가 한다 — review/qa와 같은 경계(스킬은 검증만, 영속화는 밖).
- **역할 → 모델 배정표.** 본문은 역할 슬롯(코더 단순/복잡·게이트·오케스트레이터)으로만 말하고 모델명은 `references/flow.md` 배정표 한 곳에만 둔다(신모델 대응 = 표 교체).

**실행 전 `references/flow.md`를 반드시 Read 한다 — 안 읽고 코더 스폰 금지.** 전체 절차·배정표·강도표·루프 상한·가드레일이 거기 있다. `$ARGUMENTS` = 과업 명세(+선택적 강도). 없거나 목표·완료 기준이 모호하면 1~2개만 되묻는다.

## 강도 = 라우팅 (핵심)

강도(simple/standard/critical)가 이 스킬의 핵심 판별 — **코더 슬롯 + 두 게이트 강도**를 한 번에 정한다. **정량·라우팅 서술(어느 게이트 강도로 넘기나, "언제"의 상세)은 `references/flow.md` 강도표가 정본** — 아래 표는 키워드만(rot 방지).

| 강도 | 키워드 |
|---|---|
| **simple** | 국소·저위험 |
| **standard**(기본) | 비자명 일반 변경 |
| **critical** | 동시성·kill·lifetime·보안·공개 API·마이그레이션 |

- **escalation-only:** 코더·리뷰 도중 위험 트리거(동시성·kill·lifetime·보안·공개 API·마이그레이션)를 발견하면 상위 강도로 자동 승격 + 보고한다. 시작 강도가 하한 — 임의 하향 금지.
- **이 스킬 대상이 아닌 것:** 1~2줄 사소 수정·사소한 문서(오타·서식)·조사/탐색성·스파이크는 인라인 예외 부류다. `references/flow.md §0`에서 "스킬 불필요, 인라인 + QA" 안내 후 종료한다.

## 트리거

`/implement "<과업>" [simple|standard|critical]`. 강도는 옵션 — 미지정이면 위험 영역·범위로 추정하되 **기본 standard**. 파싱·추정은 `references/flow.md §0-1`. 호출 시 **"어느 강도·어느 코더 슬롯으로 도는지" 한 줄을 사용자에게 명시**한다(예: "standard 강도 / 코더(복잡) 슬롯으로 구현합니다 — review full → qa standard").

## 프로젝트 통합 (골격/바인딩 분리)

이 스킬은 **범용 구현 오케스트레이터**다. **자체 바인딩 파일을 만들지 않는다** — 두 게이트 스킬이 이미 정본을 보유하므로 중복 SSOT를 만들지 않기 위함이다:
- **빌드·테스트 실명령** = `/qa` 스킬 바인딩(`references/bindings/<project>.md`)이 정본. 이 스킬은 재수록하지 않고 `/qa`에 위임한다.
- **코드 불변식·전용 체크리스트** = `/review code` 스킬 바인딩이 정본. 코더 지시서의 "제약(불변식·건드리지 말 것)" 컨텍스트는 프로젝트 통합 지점(CLAUDE.md·ADR)에서 추려 넣는다 — 스킬 본체에 하드코딩하지 않는다.
- **커밋·결정 기록(ADR/step-log)** = 프로젝트 관례. 이 스킬은 커밋 준비물만 반환하고 실행·기록은 넘긴다.

TDD 구체 명령·모듈 격리 하네스 형태도 프로젝트 바인딩(qa/review)에 위임한다 — 이 스킬은 "테스트 먼저 또는 함께"라는 골격만 코더 지시서에 박는다.

## ⚠️ 검증 상태 (정직)

- **실측 0회.** dogfood(실제 코드 과업 1회 완주) 전엔 staging 보류·배포 안 함(팩토리 공정 5단 — 문서 게이트 통과 ≠ 실행 검증).
- **근거 있는 것:** 이 스킬은 수동 공정(engram-dashboard "구현 실행 규약" — 코더→리뷰어→QA 서브에이전트 분리, 검증돼 운영 중)을 오케스트레이션으로 **재구성**했다 — 재구성 과정의 강제사항 무손실 여부가 dogfood 검증 대상이다. 여기에 선례 경고 방어를 더했다.
- **더한 방어 (선례 경고에서):** 무한 리뷰 루프(상한 2회) · 재귀 폭발(코더는 서브에이전트를 못 낳음) · 모델 다운그레이드(스폰 시 모델 명시 강제 — research 실측 경고 선례) · 리뷰 형식화(생산자≠리뷰어 fresh 분리).
- **미검(정직):** 이 오케스트레이션이 수동 공정 대비 실제로 결함 검출·품질을 얼마나 올리는지는 dogfood 전까지 "근거 있는 가설"로 취급한다(공정 자체는 검증됐으나 *자동화 래퍼*는 0회).
