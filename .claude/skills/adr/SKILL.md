# ADR

ADR(설계 결정 기록 — Architecture Decision Record)의 **기계적 작업을 자동화**한다. ADR은 "왜 이렇게 정했나 + 거부한 대안"을 시점 무관하게 박제해, 다음 세션이 같은 대안을 다시 꺼내는 재론(re-litigation)을 막는 영구 못이다.

이 스킬이 자동화하는 건 **기록의 기계적 일**뿐이다 — 채번 · 템플릿 파일 스캐폴드 · 인덱스 재생성 · supersede 양방향 박기 · drift lint. **결정 그 자체는 만들지 않는다.** 결정 내용·거부한 대안·근거는 호출자(메인/사용자)가 제공하고, 스킬은 그걸 받아 구조화·파일링·교차링크할 뿐이다.

## 핵심 구조 — 결정적 스크립트 + 얇은 판단 (하이브리드)

서기 작업은 **결정적 스크립트(`scripts/adr.mjs`)** 가 한다 — 채번(최대+1·재스캔)·파일 스캐폴드·인덱스 재생성·supersede 양방향 링크·형식 lint. LLM이 손으로 하면 번호 충돌·한쪽 빠뜨림·거짓 lint가 난다(1차 초안이 그래서 깨졌다 — 기계 잡일을 LLM 문서로만 명세). 스킬(LLM)은 **기계가 못 하는 일**만 한다: ① 입력 수령·검증 ② **전체 폐기 vs 부분 폐기 판단** ③ 본문 prose 채우기 ④ 결과 보고. 이 경계가 곧 스킬의 존재 이유다.

> 스크립트 실명령·경로·템플릿·인덱스 형식은 바인딩(`references/bindings/<project>.md`)이 정본. 골격에 명령을 하드코딩하지 않는다.

## 1차 축 = 오퍼레이션 (강도 아님)

review/qa는 "얼마나 깊게"라는 강도 축이 의미가 있다(깊이 트레이드오프). **ADR 기록엔 그게 없다 — 결정적(deterministic) 작업이라 "대충/철저히"의 스펙트럼이 아니다.** 의미 있는 1차 축은 *어느 기록 오퍼레이션인가*다(review의 *단계* 축과 동형). 가짜 강도를 끼우지 않는다.

| 오퍼레이션 | 무엇을 | 스크립트(기계) | 스킬(판단) |
|---|---|---|---|
| **new** | 새 결정 1건 박제 | 채번 → 스캐폴드 → 인덱스 재생성 | 본문 prose 채우기 |
| **supersede** | 결정 번복(전체/부분) — 양방향 박기 | 새 ADR + 옛 ADR 양방향 링크(full=폐기·취소선 / partial=Amends) | **전체 vs 부분 판단** + prose |
| **lint** | 정합성 점검 | 번호·양방향·상태어휘·앵커고아 검사(보고 전용) | error/advisory 해석·보고 |

오퍼레이션이 안 들어오면 대상으로 추정한다(새 결정=new / "이거 폐기"·"바뀜"=supersede / "점검"·인자 없음=lint). 추정 규칙은 `references/flow.md` §0.

## 트리거

`/adr [new|supersede|lint]`. 오퍼레이션은 옵션이다 — 미지정이면 요청으로 추정한다. 파싱·추정·기본값은 `references/flow.md` §0. 호출 시 **어느 오퍼레이션을 도는지 한 줄을 사용자에게 명시**한다(예: "new — ADR 새로 채번해 박습니다").

전체 실행 절차·오퍼레이션별 단계(판단→스크립트 호출→prose→보고)·가드레일은 `references/flow.md`(범용 골격)를 따르고, 스크립트 실명령·폴더·템플릿·인덱스 위치·앵커 명령 같은 **프로젝트 실체**는 바인딩 파일을 쓴다.

## 프로젝트 바인딩

스크립트 실명령·ADR 폴더·파일명 규약·템플릿 실체·인덱스 위치·상태 범례·코드 앵커 명령은 `references/bindings/<project>.md`. 현재 engram = `references/bindings/engram.md`. 다른 프로젝트는 같은 골격(`references/flow.md`)에 바인딩 파일만 추가한다 — 골격에 특정 경로·명령·템플릿을 하드코딩하지 않는다.

## ⚠️ 검증 상태 (정직한 표시)

ADR 스킬은 기계적 기록 자동화라 review/research 같은 "미검증 가설" 성격이 없다 — 단 보장 범위를 정직히 긋는다:

- **보장하는 것 = 기록 정합성**(채번 충돌 없음 · supersede 양방향 일치 · 인덱스↔헤더 일치 · 템플릿 형식 준수). 이건 스크립트가 결정적으로 검사·강제한다.
- **보장하지 않는 것 = 결정의 옳음.** 스킬은 결정 내용·거부한 대안·근거의 *타당성*을 판단하지 않는다 — 그건 호출자(메인/사용자)와 review 스킬(적대 검증)의 몫이다. 빈약하거나 틀린 결정도 형식만 맞으면 기록은 PASS한다.
- **LLM 환각 주의** — LLM이 본문(맥락·근거·거부한 대안)을 채울 때 없는 API·가짜 근거·지어낸 대안을 환각하는 게 검증된 실패 모드다. 거부한 대안·근거는 **사용자 제공분만** 박제하고, 채운 본문은 호출자가 **fact-check**하도록 보고한다(리서치 권고). 상세 = `references/flow.md` "LLM 환각 주의".
- 결정의 질은 ADR 스킬이 아니라 **그 앞단(prd/trd 컨설·교차검증·review)**에서 잡는다. 이 스킬은 *그 결과를 정확히 박제*하는 데만 책임진다.

## 자기개선 피드백 (임시 — 성숙하면 공용으로 추출)

이 스킬을 *쓰다가* 결함·개선점을 발견하면:
- **그 자리서 고치지 않는다** — 사용 중 오판으로 멀쩡한 스킬을 되레 망칠 수 있다.
- **작업이 끝나면 즉시** 같은 폴더의 `feedback.md`에 한 줄 기록한다(미루지 않되 작업 흐름은 끊지 않는다). 누적 — 덮어쓰지 않는다.
- 실제 반영은 **관련 주제가 다시 나올 때 사용자 승인 하에** 한다.

지금은 스킬마다 이 절을 복붙해 둔다 — 성숙하면 공용 파일 1개 + 포인터로 추출한다(트레이드오프 = 복붙 rot, 의도된 임시).
