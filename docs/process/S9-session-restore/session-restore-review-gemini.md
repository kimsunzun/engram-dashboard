---

제시된 두 문서(위즈텀 실패 분석 및 새 LLD 초안)를 바탕으로, "Claude 세션 복원은 100% 신뢰할 수 없다"는 대전제를 극단적으로 밀어붙여 허점과 결함을 찾아내는 **Adversarial(적대적) 검토**를 진행했습니다.

기존 위즈텀의 실패 원인을 잘 포착했으나, 새로운 설계 역시 Claude CLI라는 '블랙박스'에 의존하고 있어 몇 가지 치명적인 아키텍처적 함정이 숨어 있습니다.

---

## 1. AgentId = claude --session-id 통합 타당성 검토

**결론: 강한 결합(Tight Coupling)의 함정. 에이전트의 '프로필 ID'와 Claude의 '세션 ID'는 반드시 분리해야 합니다.**

**위험 시나리오 (1 에이전트 != 1 세션):**

사용자가 특정 프로젝트 환경(CWD, Env, 이름이 지정된 '백엔드 개발 에이전트')은 그대로 유지하되, 대화 컨텍스트만 새로 고침(Reset)하고 싶을 수 있습니다. 만약 ID가 하나로 통합되어 있다면, 대화를 새로 시작하기 위해 에이전트 프로필 자체를 삭제하고 새로 생성해야 하므로 UX가 극도로 불편해집니다.

**Claude CLI의 블랙박스 위험:**

만약 Claude CLI가 내부적으로 `--session-id`로 넘겨받은 UUID를 서버나 로컬 캐시(`~/.claude`)에 인덱싱할 때, 우리가 임의로 생성한 UUID v4 포맷을 거부하거나 특정 규칙(예: 자체 타임스탬프 결합 등)을 요구한다면 이 설계는 시작부터 무너집니다.

**방어적 대안:**

`AgentProfile.id`는 시스템 내부 고유 ID로 유지하고, `AgentProfile.claude_session_id: Option<String>`을 필드로 두어야 합니다. 그래야 세션이 깨지거나 포맷이 바뀔 때 프로필을 유지한 채 세션 ID만 교체할 수 있습니다.

## 2. JSON Persist 정책 검토

**결론: 원자적(Atomic) 저장 보장 부재 및 분산 상태(Distributed State)로 인한 데이터 유실 위험.**

**파일 손상(0-byte File) 위험:**

"변경 시 즉시 저장 + 앱 종료 시 저장" 정책은 앱이 쓰기 작업 도중 크래시(OS 다운, 전원 차단 등)가 나면 `agents.json` 전체가 깨져 **모든 에이전트 프로필이 증발**하는 치명적인 취약점이 있습니다. SQLite가 오버킬이라면, 최소한 `agents.json.tmp`에 쓰고 `rename`하는 **Atomic Write** 메커니즘이 강제되어야 합니다.

**프론트-백엔드 상태 불일치 (Split-Brain):**

레이아웃(슬롯)은 프론트 `slotStore`가 책임지고, 프로필은 백엔드가 책임집니다. 앱이 비정상 종료될 때, 프론트는 슬롯 매핑을 저장하지 못했는데 백엔드만 `agents.json`을 업데이트했다면, 재시작 시 "백엔드에는 프로필이 존재하지만 화면 어디에도 배치되지 않는 유령 에이전트"가 발생합니다.

## 3. 자동 재시작(Auto-Restart) 위험 분석

**결론: 독약 주문(Poison Pill)으로 인한 무한 크래시 루프 및 토큰 낭비 위험.**

**Poison Pill 컨텍스트 복원 루프:**

Claude CLI가 크래시 난 이유가 CLI 자체의 버그나 특정 대화 입력값(예: 너무 긴 코드 블록 덤프, 특정 특수문자 렌더링 깨짐 등) 때문이라면, `--resume`로 재시작하는 순간 **동일한 독약(Poison Pill) 컨텍스트를 다시 읽어와 또다시 크래시**가 납니다. `max_retries`가 있더라도 그동안 CPU와 네트워크 자원을 낭비하게 됩니다.

**인증(Auth) 만료로 인한 좀비화:**

Claude 세션이 만료되어 세션이 종료된 경우, 자동 재시작이 발동하면 계속해서 인증 에러를 뿜으며 재시작을 반복합니다. CLI의 Exit Code만으로는 이것이 '프로세스 크래시'인지 '인증 만료'인지 구별하기 어렵습니다.

---

## 4. Resume 실패 시 Fallback UX 검토

**결론: "기억 상실"의 무묵인(Silent Error)은 최악의 UX입니다.**

현재 설계 안:
```
claude --resume <id> 실패
  ↳ 100% 불신 모드 발동
  ↳ claude --session-id <id> (Fresh)로 조용히 재시작
```

**사용자 기만 위험:**

사용자는 이전까지 1시간 동안 진행한 복잡한 아키텍처 컨텍스트가 살아있다고 믿고 "방금 말한 구조대로 코드 짜줘"라고 명령할 것입니다. 하지만 시스템이 조용히 Fresh 세션으로 돌렸기 때문에, Claude는 완전히 뚱딴지같은 소리를 하거나(환각), 아무것도 모른다고 답할 것입니다.

**방어적 대안:**

Fallback이 발생하면 반드시 UI 상에 명시적인 경고(Warning Signal)를 띄워야 합니다.

> ⚠️ "이전 세션 복원에 실패하여 새로운 대화 세션으로 전환되었습니다. 기존 대화 맥락이 유실되었을 수 있습니다."

## 5. Spike 단계에서 반드시 검증해야 할 Claude CLI 함정 (추가 제안)

현재 설계의 6번 항목(Spike 필요)에 더해, **이것이 검증되지 않으면 구현이 불가능한** 핵심 질문들입니다.

**동시성(Concurrency) 제약:**

동일한 세션 ID(`--session-id <id>`)를 가진 Claude 프로세스를 **동시에 두 개 이상의 PTY 패인에서 띄우면 어떻게 되는가?** 하나가 차단되는가, 아니면 서버 측 세션 데이터가 덮어씌워져 상호 오염(Data Corruption)이 발생하는가?

**비대화형(Non-interactive) 환경에서의 에러 덤프:**

`--resume` 실패 시, Claude CLI가 에러 메시지를 `stderr`로 뱉고 즉시 종료(Exit Code != 0)되는가? 아니면 TUI 화면 내부(PTY)에 "Session not found"라는 글자만 렌더링한 채 **사용자 입력을 기다리며 대기(Hang)하는가?** 후자라면 백엔드 드레인 스레드는 실패를 감지하지 못합니다.

**네트워크 타임아웃 및 세션 유효기간:**

로컬 `~/.claude`에 세션 정보가 있어도, Claude 서버 측에서 해당 세션을 만료시켰을 때 CLI가 어떻게 반응하는가?

---

## ⚖️ 최종 판정: 구현을 시작해도 되는가?

**진행 판정: "백엔드 코어 구현은 보류(No-Go), Spike 검증은 즉시 시작(Go)"**

아직 설계만으로는 실제 구현에 착수하기에 **리스크가 너무 높습니다.** Claude CLI의 동작이 완전히 불확실한 상태에서 Rust 구조체와 매니저 스레드부터 짜기 시작하면, Spike 결과에 따라 아키텍처 전체를 갈아엎어야 할 수 있습니다.

### 구현 시작 전 반드시 확정해야 할 위험 사항 매트릭스

| 위험 요소 | 확정되어야 할 사항 | 미확정 시 발생하는 최악의 상황 |
|---|---|---|
| `--session-id` 임의 지정 가능 여부 | Claude CLI가 우리가 생성한 UUID 포맷을 온전히 받아들이는지 실측 | 설계 0번(핵심 결정)이 통째로 무력화됨 |
| Resume 실패 시의 시그널 | 실패 시 정확한 Exit Code 또는 Stderr 키워드 확보 | Fallback 메커니즘이 작동하지 않고 무한 대기(Hang) 상태 유입 |
| Atomic Write 미비 | 파일 쓰기 중 크래시 대책 (`tempfile` -> `rename` 패턴 적용 여부) | 사용자 설정 데이터 전체 유실 및 앱 초기화 |

**추천 액션 플랜:**

우선순위 6번의 **Spike 3종 + 추가 제안 3종**을 Windows Powershell/PTY 모킹 환경에서 한 시간 이내로 빠르게 돌려보고, 그 결과값(Exit Code, TUI 렌더링 문자열)을 로그로 확보한 직후에 백엔드 데이터 모델링 코딩을 시작하십시오.
