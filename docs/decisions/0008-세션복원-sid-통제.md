# ADR-0008: 세션 복원 — 우리가 sid 통제, 추적 파일은 best-effort

- 상태: 확정 (S9, 2026-06-15 데몬 수명 맥락 보강)
- 관련: CLAUDE.md S9 메커니즘 · `backend/claude.rs` · `profile.rs::ProfileRegistry` · `session_tracker.rs` · ADR-0015(데몬 수명)

## 맥락
claude 세션을 재시작 후 무손실 복원해야 한다. session id(sid)를 누가 통제하느냐가 복원 정확성을 좌우한다.

## 결정
- spawn 시 **`--session-id <uuid>`로 우리가 sid를 발급·persist** → 재시작 시 `--resume <uuid>`로 복원.
- `/clear`로 claude가 sid를 바꾸면 `~/.claude/sessions/<pid>.json`을 폴링해 따라잡아 **즉시 persist**(best-effort).
- resume이 조기 종료(윈도 3s 내 terminal)하면 실패로 보고 새 sid로 fresh fallback(종점 Failed, 재귀 금지).

## 거부한 대안
- **claude 자동 sid에 의존** — 우리가 통제 못 해 복원 시점에 어느 세션인지 특정 불가.
- **`sessions/<pid>.json`을 1급 의존으로** — 못 읽으면 복원 자체가 실패. 그래서 best-effort로 강등.

## 근거
복원 정확성은 **우리가 통제하는 sid에만** 의존한다. 추적 파일은 못 읽어도 무손상 강등(추적만 끔). S9 fable 리뷰 M-1: 성공한 resume은 TUI라 윈도 내 종료하지 않으므로 윈도 내 terminal은 code 무관 실패 신호.

## 영향 / 불변식
- **`sessions/<pid>.json`으로 기능을 *확장*하지 말 것** — best-effort 등급 고정.
- resume 조기 종료 윈도 = 3s(`EARLY_EXIT_WINDOW`).

## 데몬 수명 맥락 보강 — 부팅 복원 정책 (2026-06-15, ADR-0015, 사용자 결정)
데몬이 persist-until-kill(ADR-0015)이므로 **복원은 "데몬 콜드 부팅 시"에만** 발동한다 — PC 재부팅·명시적 `shutdown` 으로 데몬이 실제 죽었다 살아날 때뿐. 앱 창 여닫기·재연결로는 데몬이 안 죽으니 claude 세션은 데몬 안에서 **계속 살아 돌아가고 복원이 발동하지 않는다**(평상시 매번 리프레시 없음 — Emacs 데몬/Zellij 서버가 프로세스를 살려두는 것과 동형). 이것이 직전 세션의 "콘솔 닫음→데몬 죽음→재복원" churn 을 구조적으로 제거한다.

**부팅 복원 정책 = 3모드 선택 옵션(global) × per-profile `auto_restore` 플래그(eligibility):**
- 플래그 = "이 에이전트를 복원 대상에 넣을까"(기존), 모드 = "대상들을 부팅 때 어떻게 할까"(신규).
- 모드: ① 무조건 복원(자동) ② 무조건 안 함 ③ **물어보기**(콜드 부팅 후 UI 첫 진입 시 "N개 복원?"). **기본 = ③**(콜드 부팅은 드무니 1회 prompt 비침습 + claude N개 갑작스런 spawn 놀람 방지; 무인/파워유저는 ①로 전환). §5 상 모드 전환은 LLM 도 호출 가능한 설정 핸들로 노출.

**부팅 복원은 세션당 1회성 — 별도 카운터 불필요.** 기존 사다리(`--resume` → 3s 내 조기종료면 fresh 1회 fallback → 그것도 실패면 종점 Failed, 재귀 금지)가 그 자체로 가드. 실패해도 트리에 Failed 로 1회 표시되고 멈춘다.

**구분(혼동 금지):** "살아 돌던 에이전트가 런타임에 죽었을 때 자동 되살림"은 **별개 기능(자동 재시작)** 이고 backoff+최대횟수 제한이 필요하다 → **TODO(게이트, 코어 안정 후).** 부팅 복원(1회성)과 섞지 말 것.
