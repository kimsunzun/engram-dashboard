# ADR-0014: 오케스트레이션 참조 후보 (설계 시 고려)

- 상태: **제안** (2026-06, 후보 기록 — 실제 채택은 오케스트레이션 설계 착수 시 결정)
- 관련: ADR-0013(데몬 3대장)과 상보 · CLAUDE.md 참조 구현 · 로드맵 "메시지 시스템/자동 재시작"
- **심화 조사(2026-07-12):** `docs/research/orchestration-survey-2026-07-12.md` — 이 후보들을 층별로 서베이 + engram 기존자산 매핑 + 옵션셋 3개 + Codex 적대 리뷰. 설계 착수(TRD) 시 그 옵션에서 사용자 결정 → 이 ADR을 확정으로 갱신하거나 후속 ADR.

## 맥락
engram은 터미널 데몬(ADR-0013) 위에 **다수 에이전트 조율(감독·재시작·메시지 패싱)** 층으로 확장될 예정이다. 그 설계는 "한 번 정하면 오래 가는" 결정이라, 바닥부터 짜기 전에 검증된 선행 사례를 후보로 박아둔다. engram 제약 = Rust + 독립 프로세스(PTY) 에이전트 + 감독 + 메시지.

## 후보 (층별)

**A. 감독·내결함성 패턴 (engram 핵심 — 죽음 감지/재시작)**
- **Erlang/OTP supervision tree (let it crash)** — supervisor가 worker 감시→크래시 시 재시작, 프로세스 격리. engram 감독 모델과 동형. *개념 1순위.*
- **Ractor (Rust)** — OTP식 액터+supervision의 Rust 구현. 같은 스택이라 이식 가능. (대안: Actix)

**B. 내구성 오케스트레이션 (상태 안 잃기)**
- **Temporal** — durable execution 표준(영속 상태·재시도·heartbeat).
- **Restate / Inngest** — 가벼운 대안(저널링 재연 / 이벤트 step 재시도).

**C. 에이전트 오케스트레이션 프레임워크 (조율 로직 — 패턴 차용용, Python 중심)**
- **LangGraph** — 그래프+체크포인트+롤백+HITL. 현재 채택 최다.
- **Microsoft Agent Framework** — AutoGen/AG2 후속(1.0 GA 2026-04). 이벤트 기반·GroupChat.

**D. 인터-에이전트 통신 프로토콜**
- **A2A (Agent2Agent, Google)** — 에이전트 간 표준 메시지 프로토콜. "메시지 시스템" 로드맵 시 참조.

## 거부/보류한 대안
- Ray(분산 액터 병렬) · Akka(JVM 액터) · Google ADK · OpenAI Agents SDK · CrewAI — honorable mention. engram 규모(로컬·소수 에이전트)엔 과하거나 스택 불일치. 필요 시 재검토.

## 근거
engram은 *터미널/프로세스 에이전트 + 감독*이라 **A(Erlang/OTP·Ractor)가 가장 직결**. B는 상태 영속, C는 조율 로직(이식 아닌 패턴 차용), D는 통신 표준. LangGraph류는 Python 중심이라 코드 이식 대상 아님.

## 영향 / 불변식
- 오케스트레이션 설계 착수 시 이 후보들을 우선 검토하고, 채택이 확정되면 이 ADR을 **확정**으로 갱신하거나 후속 ADR로 결정 기록.
- 차용 = 패턴이지 코드 복붙 아님(복사 시 라이선스 법무 확인).
