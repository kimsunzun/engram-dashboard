# ADR-0012: 테스트 전략 — 모듈 격리 하네스 + TDD

- 상태: 확정 (S3~S10 누적, S12에서 강제화)
- 관련: CLAUDE.md 구현 실행 규약·빌드 명령 · `examples/{headless,transport_smoke,session_smoke}` · ADR-0003

## 맥락
모듈을 독립적으로 검증할 수 있어야 회귀를 빨리 잡고, 동시성·lifetime 버그를 어느 모듈에서 났는지 격리할 수 있다. LLM 코딩은 그럴듯하지만 틀린 코드를 만들기 쉬워, 통과/실패를 자동으로 가르는 그물(테스트)이 먼저 있어야 한다.

## 결정
1. **모듈 격리 하네스** — 모든 모듈은 외부 의존(Tauri/네트워크/실제 프로세스)을 seam으로 끊어 **단독 실행 가능**해야 한다.
   - 코어: Noop/테스트 sink로 `examples/headless`(manager 전체, 프론트 없이).
   - transport/session: `examples/transport_smoke`·`session_smoke`(manager 없이 직접 실측).
   - 데몬 모듈: integration test + harness bin으로 한 번에 돌도록 모은다.
2. **TDD** — 기능 단위로 **테스트를 먼저(또는 함께) 작성**하고 누적한다. `cargo test`(workspace 루트) 한 번으로 전 모듈을 회귀 검증.
3. **코드 통과 ≠ 완료** — QA는 build/test + GUI 실측(`scripts/cdp.mjs`)까지 포함해야 미완 해제.

## 거부한 대안
- **통합 테스트만(앱 전체 띄워 검증)** — 느리고, 실패 시 어느 모듈 문제인지 격리 안 됨. 동시성 버그 재현 어려움.
- **테스트를 나중에 몰아서** — 회귀를 그때그때 못 잡고, LLM 환각 코드가 통과해 누적됨.

## 근거
`OutputSink`/`StatusSink` seam(ADR-0003)이 코어의 격리 실행을 이미 가능케 한다. examples/headless·smoke가 실제로 프론트·manager 없이 코어/transport를 단독 검증하고 있음(S3·S10 회귀 0 확인).

## 영향 / 불변식
- **새 모듈은 격리 하네스를 동반**한다(`#[cfg(test)]` 단위 + 필요 시 example/harness bin).
- 외부 의존을 직접 호출해 격리 불가능하게 만드는 코드는 이 ADR 위반 — seam으로 끊는다.
