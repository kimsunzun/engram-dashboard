# ADR-0015: 데몬 수명 = persist-until-kill, 콘솔 = detachable 뷰어

- 상태: 확정 (2026-06-15, 근거: prior-art 조사 tmux/Zellij/Emacs/mosh/dtach + 직전 세션 churn 사건)
- 관련: CLAUDE.md §5(LLM-우선 제어)·"데몬/전송 앵커" · ADR-0013(데몬 참조 3대장) · ADR-0008(복원, 보강) · `src-tauri/src/discovery.rs` · `crates/engram-dashboard-daemon/`

## 맥락
직전 세션에 데몬 콘솔(빈 창)을 데몬 생명줄로 묶어둔 탓에, 콘솔 창을 닫으면 → 데몬이 죽고 → 앱이 재discover → 재spawn → 세션 복원이 반복되는 **churn 루프**를 겪었다. "데몬은 언제 살고 언제 죽는가"와 "콘솔 창과 데몬의 관계"를 명확히 박아야 한다.

prior-art 조사 결론: tmux·Zellij·Emacs 데몬·mosh·dtach **전부 "PTY 를 쥔 프로세스"와 "그것을 보는 뷰어/콘솔"을 분리**하고, 런칭 터미널에서 데몬화(daemonize)해 창 수명에 묶이지 않는다. 우리가 겪은 "콘솔 창이 데몬 수명을 소유"하는 구조는 어느 도구에도 없다 — 그게 버그의 정체.

## 결정
1. **데몬 수명 = persist-until-kill.** 에이전트가 0이어도 백그라운드로 살아있는 게 우리 전제다. 데몬은 **명시적 `shutdown` 명령(사람 또는 LLM 호출)으로만 죽는다.** tmux식 "0 에이전트 + 0 클라이언트면 자동종료" 옵션은 **두지 않는다**(불필요한 복잡도 — 단일 정책으로 단순화).
2. **데몬은 windowless 로 spawn.** Windows WMI `Win32_Process.Create` 시 `CREATE_NO_WINDOW`/`DETACHED_PROCESS` 로 콘솔 창 자체를 할당하지 않는다 → "닫으면 데몬이 죽는 창"이 존재하지 않는다.
3. **콘솔 = detachable 뷰어 클라이언트.** 콘솔은 데몬의 생명줄이 아니라 **살아있는 데몬에 옵션으로 붙었다 떼는 뷰**다(tmux attach/detach, Emacs `emacsclient -c`). 콘솔을 닫아도 데몬은 생존, 다시 열면 재attach. 떠 있을 때의 용도는 **UI 없는 디버그/명령 콘솔창** — 디버그 메시지 표시 + 데몬에 직접 명령 송신(REPL성). 콘솔(및 모든 뷰어)은 **데몬을 spawn 하거나 kill 할 수 없다**(읽기·명령만; 수명 제어는 별도 `shutdown` 핸들로만).
4. **ensure-on-open.** 앱이 열릴 때 데몬을 확보한다 — 살아있으면 attach, 없으면 start 후 attach(Emacs `emacsclient --alternate-editor=""` 패턴). 앱을 닫으면 WS 연결만 끊기고 데몬은 생존.

## 거부한 대안
- **tmux `exit-empty` 식 자동종료(0 에이전트+0 클라 → 자동 exit, grace window)** — 자원회수 이점은 있으나 "백그라운드 상시 생존"이라는 우리 전제와 배치되고, grace 타이머·agents×clients 사분면 등 테스트 표면만 늘린다. 사용자 명시 결정으로 옵션조차 두지 않음.
- **mosh식 타임아웃 단독(끊기면 N초 후 exit)** — 로컬 IPC라 불안정 네트워크 회복 이점이 없고, 유휴 후 "데몬 사라짐→재spawn→재복원" churn 을 재도입한다. grace debounce 아이디어만 (향후 필요 시) 차용.
- **콘솔을 데몬의 부모/소유 창으로(현행)** — 직전 churn 의 원인. 폐기.
- **콘솔/뷰어에 데몬 spawn·kill 권한 부여** — 손발/두뇌 분리(§5) 위반. 수명 제어는 명시 핸들 1곳으로 모은다.

## 근거
- prior-art: ensure-daemon-on-open ⇐ Emacs `emacsclient --alternate-editor=""`; persist-on-detach ⇐ Zellij `ActiveDetached`/Emacs 데몬; "뷰어는 socket 클라일 뿐, PTY 쥔 프로세스와 분리" ⇐ dtach/abduco.
- §5 LLM-우선 제어: 데몬 수명이 콘솔 창의 암묵적 동작이 아니라 **명시적 `shutdown` 명령 핸들**이 되어야 사람 클릭과 LLM 호출이 같은 진입점을 거친다.
- 직전 세션 churn 이 "콘솔=생명줄" 설계의 실패 증거.

## 영향 / 불변식
- **콘솔/뷰어는 데몬을 spawn·kill 하지 못한다** — 데몬 수명 제어는 명시 `shutdown` 핸들(사람/LLM)로만. 어기면 churn 재발.
- **데몬 spawn 은 windowless** — 콘솔 창에 묶이는 경로를 만들지 말 것.
- **자동종료 정책 없음** — "유휴면 데몬 끄자"는 제안이 다시 나오면 이 ADR을 폐기(Superseded)로 새 번호에 기록할 것(덮어쓰기 금지).
- 복원 정책은 ADR-0008 참조(데몬 콜드 부팅 1회).
