# ADR-0026: 트레이/프로세스 토폴로지 재결정 — 트레이를 Tauri 앱에 통합(2프로세스), 데몬 별도 (ADR-0023 폐기)

- 상태: 확정 (2026-06-19, dashboard9 세션 — 사용자 결정 + `/consult` 3종 교차검증 job `20260619-215927-consult-tray-app-merge`)
- 관련: ADR-0023(폐기됨, 본 ADR이 대체 — Supersedes ADR-0023)·ADR-0024(데몬 소유·종료 — 본 ADR이 불변식 재확인)·ADR-0022(통합 command registry)·ADR-0025(부팅 데몬 ensure)·ADR-0013(tmux 참조)·CLAUDE.md §5(LLM 제어) · `docs/process/step-log.md` dashboard9
- 범위: daemon 모드에서 트레이·UI·데몬을 어떤 프로세스로 나누나. ADR-0023의 "3프로세스(순수-Rust 별도 트레이)" 결정을 뒤집는다.

## 맥락
ADR-0023은 트레이를 **별도 순수-Rust 프로세스**로 분리했다(트레이 / Tauri UI / 데몬 = 3프로세스). 분리의 **유일한 정당화**는 "로그인 상주 트레이를 WebView 메모리(~150MB) 없이 항상 띄운다"였다.

S13 sub-step 2를 구현하며 "트레이 → UI 창 열기/닫기"를 설계하던 중, 별도 트레이 프로세스가 별도 UI 프로세스의 창을 제어하려면 **크로스-프로세스 IPC**가 필요함이 드러났다. 이를 OSS로 조사한 결과(VS Code·Docker·Tailscale·Syncthing·Zellij): **트레이+GUI를 별도 프로세스로 쪼갠 선례가 사실상 없다**(전부 한 프로세스, 혹은 트레이=데몬 바이너리). 동시에 분리의 명분(WebView 메모리)은 이 제품 맥락에서 무너졌다 — 타깃이 개발자 워크스테이션(RAM 충분)이고, 이미 "UI 닫기=hide(창 떠있는 동안 WebView 상주)"를 수용해 별도 트레이가 절약하는 구간은 "부팅 후 UI를 한 번도 안 연 시간"뿐이다.

## 결정
**2프로세스로 통합한다.**
- **로컬 앱(Tauri)** = 시스템 트레이 + 메인 창을 **한 프로세스**가 소유(Tauri v2 네이티브 트레이). 로그인 상주(autostart). 트레이 메뉴(데몬 켜기/끄기·UI 보이기/숨기기·완전 종료)는 전부 이 앱 안.
  - **창 보이기/숨기기 = 프로세스 내부 `window.show()/hide()`** — 크로스-프로세스 IPC 불필요.
  - **X = hide**(prevent_close), 진짜 종료는 트레이 "완전 종료"만.
- **데몬** = 별도 백엔드(에이전트 호스트). detached, 앱 생사와 **독립 생존**(ADR-0024 그대로). 앱은 데몬을 ensure/연결만 하고 **소유하지 않는다**.
- **데몬 선택은 client(앱) 측 profile/context.** client:daemon = 1:N. 로컬 데몬만 lifecycle-managed, 원격 데몬은 connect/disconnect 대상(향후).

### 유지되는 불변식 (ADR-0023에서 살아남는 진짜 자산 — 통합과 무관하게 보존)
1. **두 평면 분리 (load-bearing).**
   - **로컬 제어 평면:** 데몬 lifecycle(켜기/끄기/health)·앱 창·autostart 제어. 전송 = named pipe/unix socket(OS user 경계 인증) 또는 통합 앱이면 **프로세스 내부 함수 호출**. 외부(LLM/CLI) 제어 표면으로만 IPC 노출.
   - **원격 데이터 평면:** 에이전트 I/O(PTY 출력/입력·spawn/kill/resize·attach/replay). 전송 = WS(embedded→local WS→remote WSS), 원격은 token/mTLS 필수.
   - **로컬 창/트레이 제어를 원격 데이터 채널로 라우팅 금지.** "창 보이기"가 원격 데몬으로 나가는 일은 없다.
2. **데몬은 앱의 OS 자식이 아니다 (ADR-0024 C1 재확인).** detached spawn, Windows Job `KILL_ON_JOB_CLOSE`에 **절대 데몬을 넣지 않는다**(`CREATE_BREAKAWAY_FROM_JOB`/별도 Job). 어기면 앱 크래시·업데이트·재시작 때마다 진행 중 에이전트 PTY가 전멸한다.
3. **데이터 평면은 transport seam(`agentClient`) 뒤 교체.** embedded/local WS/remote WSS가 같은 인터페이스. 연결 *선택*은 client(profile), 연결 *구현*은 seam 뒤(프론트가 소켓을 직접 쥐는지는 impl 디테일이지 계약 아님).
4. **§5 — 트레이 액션 = Tauri command.** 네이티브 트레이 팝업은 WebView DOM이 아니라 cdp가 직접 클릭 못 한다. 트레이 동작을 전부 command/store(ADR-0022 registry)에 연결해 LLM이 같은 command surface로 호출·검증한다.

## 거부한 대안
- **ADR-0023의 3프로세스(순수-Rust 별도 트레이) 유지** — 분리 명분(WebView 메모리)이 개발 워크스테이션 + hide 수용으로 붕괴. 별도 트레이는 WebView가 없어 §5 LLM 제어/cdp 표면 밖(원칙 위반 대기)이고, S13에서 graceful 끄기·아이콘 race·콘솔 가시성 등 크로스-프로세스 race 부채를 이미 만들었다. 선례도 없음. `/consult` 3종 전원이 통합 권고. 유지 정당 시나리오(트레이가 부팅 직후부터 떠야 + GUI 거의 안 염 + RAM 극빈)는 이 제품과 정반대.
- **데몬을 앱과 같은 Job Object에 묶고 `KILL_ON_JOB_CLOSE`로 동반 종료**(Gemini 제안) — judge가 기각. "고아 데몬 방지"라는 부차 문제를 풀려다 **"데몬은 앱과 독립 생존"(ADR-0024 C1·핵심 요구) 정면 위반**. 앱이 죽으면 에이전트가 다 죽는다. 데몬은 detached+breakaway로, 고아는 lockfile/health 재발견 + idle self-shutdown으로 푼다.
- **프론트(WebView/JS)가 데몬에 직접 연결, 로컬 Rust transport seam 우회**(Gemini 제안) — 확정 아키텍처(`agentClient` seam, ADR-0011/0020)와 충돌. "누가 어느 데몬에 붙나"(client 선택)와 "소켓 구현 위치"(seam 뒤)는 다른 층위. 선택은 client, 구현은 seam 뒤로 유지.
- **2프로세스인데 트레이↔UI를 named pipe로**(통합 전 검토안 B1) — 한 프로세스로 합치면 IPC 자체가 함수 호출로 사라져 불필요. 통합이 더 단순.

## 근거
- `/consult` 블라인드 judge: 3종 전원 롤백(3→2) 찬성. 쟁점 판정 — 데몬 lifecycle은 detached+breakaway(GPT·Claude 옳음, Gemini의 KILL_ON_JOB_CLOSE 치명 오류), 데이터 평면은 transport seam(Claude 정합), WebView2 hidden은 "상주·비용 존재·실측 필요"(GPT·Claude 옳음, Gemini의 자동 suspension 낙관 단정 기각). 가장 신뢰 = Claude.
- OSS 코드/문서 조사(서브에이전트 3종): 트레이+GUI 별도 프로세스 선례 없음. 백엔드 선택은 항상 client측(Docker context·VS Code remoteAuthority·Syncthingtray connection config, 1:N). 로컬 제어를 원격 백엔드로 라우팅하는 사례 없음.

## 영향 / 불변식
- `engram-tray-host` crate(별도 프로세스)는 **제거**한다. 살릴 로직(MenuAction 의도 모델·icon 상태 매핑·to_grayscale)은 src-tauri 트레이 모듈로 이관. single-instance는 `tauri-plugin-single-instance`로 대체.
- 진입점 = 통합 앱. **autostart**(`tauri-plugin-autostart`, 부팅 시 `--hidden`으로 트레이만), **single-instance**(hidden 창 raise = `show()`→`unminimize()`→`set_focus()` + Windows focus-stealing 회피), **updater**(재시작 중 데몬 생존·재연결) 표면이 전부 이 앱에 묶인다.
- "UI 닫힘 + 트레이만 생존"은 한 프로세스로 표현(창 hide 또는 destroy, tao 이벤트 루프 잔존). Windows 전용이라 macOS last-window-quit 정책 무관.
- **구현 전 spike 실측 4종:** ① WebView2 idle/hidden 메모리, ② hidden 창 raise(focus stealing), ③ updater 재시작 중 로컬 데몬 reconnect, ④ 앱 크래시 후 기존 데몬 재발견·재attach. 통과 시 통합 토폴로지 확정.
- ADR-0024(데몬 소유·종료)·0025(부팅 ensure)는 유효 — 이제 *통합 앱*이 그 ensure/stop 주체. ADR-0022 registry에 트레이 command가 합류.
