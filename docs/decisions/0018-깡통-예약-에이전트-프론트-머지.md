# ADR-0018: 깡통(예약) 에이전트 = 저장 프로필의 트리 표현 + 더블클릭 활성화 (프론트 머지, 백엔드 무변경)

- 상태: 확정 (2026-06-16, 근거: dashboard4 세션 대화 — 사용자 개념 제시 + 적대적 검토 후 위임)
- 관련: ADR-0016(수명 모델: 저장=살림·spawn 분리) · ADR-0007(epoch 재구독) · ADR-0017(셸 비영속) · `agent/profile.rs` · `commands/profile.rs` · `src/components/agent/AgentTree.tsx` · `src/store/agentStore.ts`
- 범위: "기획서를 N개 에이전트로 분배하되 한꺼번에 spawn하지 않고 깡통으로 예약 → 사용자가 더블클릭으로 하나씩 활성화"라는 UX의 구현 방식 결정. **프론트엔드 전용, 백엔드/프로토콜 변경 0.**

## 맥락
기획서 보고 claude 에이전트 30개를 예약해두되, claude 다수 동시 실행을 피하려고 한꺼번에 띄우지 않는다. 깡통(빈 슬롯)으로 저장만 해두고 필요할 때 더블클릭으로 활성화한다. 조사 결과 **예약→활성화 배관은 이미 존재**한다: 백엔드 `create_claude_profile`(spawn 없이 upsert만)·`spawn_profile`(저장 프로필로 spawn)·`list_profiles`·`delete_profile` 완비, agentClient/embeddedClient 인터페이스도 완비. 갭은 **프론트 트리가 실행 중 에이전트(AgentInfo)만 렌더**하고 저장된 프로필을 안 보여주는 것뿐.

## 결정
1. **깡통 = 저장된 `AgentProfile`(spawn 안 함).** 이미 있는 `CreateProfile`/`spawn_profile` 배관을 재사용한다. 신규 백엔드 배관·프로토콜 variant 추가 없음.
2. **"Reserved/대기" 상태는 프론트가 합성한다** — `listProfiles()` ∖ `agents[]`(실행 중). 백엔드 `AgentStatus` enum·protocol에 비실행 상태를 추가하지 **않는다**.
3. **트리 = 프로필 ∪ 실행중.** 한 에이전트는 실행중 XOR 예약 중 정확히 한 통에만 든다(중복 없음). 예약 노드 더블클릭 → `spawnProfile(id, resume=false)` → 실행중 전환. 실행중 노드는 기존 동작(선택/우클릭) 유지. 프로필 없는 ad-hoc 셸(SpawnByCwd)도 `agents[]`에서 계속 표시.
4. **예약 생성 = `auto_restore=false`.** 부팅 자동 spawn 제외(restore_all은 auto_restore만 띄움) → 재부팅해도 깡통으로 남는다. 사용자가 원하면 `set_profile_auto_restore`로 토글.
5. **프로필 변경 이벤트가 없으므로** create/delete/activate 후 `listProfiles` refetch + 부팅 시 1회 로드로 store를 갱신한다. 실행중 전환은 기존 `agent-list-updated`가 처리.
6. **AgentSession은 건드리지 않는다** — 빈 세션을 미리 만들지 않는다.

## 거부한 대안
- **빈 AgentSession을 미리 만들어 세션맵에 보관(hollow + `IsValid()` 가드).** AgentSession은 `Box<dyn AgentTransport>`(살아있는 PTY 핸들)+`Arc<OutputCore>`(돌고 있는 pump)를 소유해 빈 상태가 설계에 없다(`session.rs:39-41`). 결정타는 **직렬화 불가** — 깡통 영속이 세션 그릇에선 원천 불가능해 결국 profile까지 따로 둬야 하므로 오히려 2벌이 된다("한 군데 관리"라는 동기와 정반대). 그 외 finalize-1회·epoch 재구독 충돌, fill 도중 TOCTOU. IsValid 가드는 read/write만 막고 직렬화·상태주인 문제를 못 푼다. (Unreal UObject 세계의 표준 패턴이나, Rust는 invalid를 타입으로 제거하는 걸 선호하고 직렬화 요구가 언어 무관하게 막는다.)
- **백엔드 `AgentStatus::Reserved` 추가.** 프로필/에이전트 2목록으로 이미 표현 가능 → 중복. `AgentStatus`는 실행 세션 수명(pump 단독 terminal)을 표현하는 자리라, 비실행 상태를 섞으면 의미가 흐려진다.

## 근거
- 조사(Explore): 백엔드·agentClient 인터페이스 완비, 갭은 프론트(store에 profiles 필드 없음, eventBus가 프로필 미구독, 트리가 실행중만 렌더). 즉 최소 변경면이 프론트.
- LLM-우선 제어(§5) 무손상: 예약·활성화 제어 표면(`create_claude_profile`/`spawn_profile`)은 이미 invoke로 LLM 제어 가능. 이번 프론트 UI는 같은 핸들을 사람이 흔드는 보조 입력일 뿐.

## 영향 / 불변식
- **백엔드 회귀 0**(무변경) — 코어/프로토콜 테스트 영향 없음.
- 깡통의 영속 단위 = `AgentProfile`(agents.json). 셸은 여전히 비영속(ADR-0017).
- 활성화 spawn은 epoch를 발급 → 프론트 `[agentId, epoch]` 재구독(ADR-0007) 그대로.
- 끄기 = 삭제(ADR-0016). 예약 깡통 삭제 = `delete_profile`.

## 후속 / 알려진 한계 (구현 후 reviewer-deep + QA 실측으로 확정)
- **활성화한 깡통이 종료되면 자동으로 깡통 복귀 안 됨 + kill 전엔 재활성화 막힘.** spawn 가드(`manager.rs:90-100`)가 sessions 맵 존재만 보고(생사 무관), 종료 세션을 reap하는 `done_rx` supervisor가 미구현(ADR-0016 §37 "승인 대기")이라 `spawn_profile` 재호출이 "already running"으로 거부된다. 현재 경로: 종료 노드 우클릭 kill → 세션 제거 → 깡통 복귀 → 재활성화. **이건 ADR-0016이 파킹한 런타임 죽음 관측/재시작/reap 결정에 속한다** — 이번 프론트 범위에서 우회하지 않고, 현 동작을 머지 테스트로 박제(회귀 가드)했다.
- **외부/LLM이 데몬 명령으로 만든 프로필은 사람 트리에 라이브 반영 안 됨.** profile 생성/삭제는 `agent-list-updated` 같은 broadcast 이벤트를 발행하지 않아, 같은 클라가 직접 `refreshProfiles()`를 부를 때만 갱신된다(Sidebar 생성 폼은 호출함 → 사람 경로는 즉시 반영 OK). LLM-우선 제어(§5)를 살리려면 데몬이 `ProfileListUpdated`를 broadcast해야 한다 — 백엔드/프로토콜 후속(이번 범위 밖).
