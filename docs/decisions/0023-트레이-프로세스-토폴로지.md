# ADR-0023: 트레이/프로세스 토폴로지 — 순수-Rust tray-host + detached 데몬 + UI(창 hide), 3프로세스

- 상태: **폐기 (Superseded by ADR-0026)** — 2026-06-19 dashboard9에서 2프로세스(트레이를 Tauri 앱에 통합)로 뒤집음. 분리 명분(WebView 메모리)이 개발 워크스테이션 + hide 수용으로 붕괴, 별도 트레이가 §5 LLM 제어 표면 밖이고 선례도 없음(`/consult` 3종 전원 통합 권고, job `20260619-215927-consult-tray-app-merge`). **단 본 ADR의 "두 평면 분리"·"데몬 detached/breakaway"·"X=hide"는 ADR-0026이 그대로 계승.** ~~확정 (2026-06-18, dashboard6 세션 — 사용자 결정 + `/consult` job `20260618-151957-consult-tray-daemon-topology`)~~
- 관련: CLAUDE.md §5(LLM 제어)·ADR-0020(단일 프로토콜+carrier)·ADR-0021(데몬 on-demand 수명)·ADR-0024(데몬 소유·종료 모델)·commit `664d629`(main 닫으면 앱 종료 — 본 ADR이 조정) · `docs/process/S13-tray-lifecycle/tray-topology-trd.md`
- 범위: daemon 모드에서 트레이·UI·데몬을 어떤 프로세스로 나누고, 창 닫기·트레이 메뉴 동작을 어떻게 정하나.

## 맥락
"UI를 닫아도 에이전트(작업)는 살아있고, 다시 열면 재연결되며, 트레이는 상시 떠서 종료를 담당한다"는 요구. 이를 위해 (a) 트레이를 어느 프로세스가 소유하나, (b) 창 닫기 의미, (c) 상시 프로세스 구성을 정해야 했다. 데몬 분리 인프라(WS·ConnectionCore·wsTransport·daemon crate)는 ADR-0020/0021로 이미 존재.

## 결정
**상시 3프로세스. 단 tray-host는 WebView 없는 순수 Rust 트레이 유틸.**
- **tray-host**(신규, tao + tray-icon, WebView 없음): 시작프로그램 상주 런처 + lifecycle owner. 데몬·UI를 띄우고/감추고/없애고, 상태 관찰, 전체 종료 coordinator.
- **daemon**(기존 crate): 에이전트 코어 보유, 헤드리스, detached(ADR-0024).
- **UI 앱**(기존 Tauri): React 창, 순수 I/O. **X 버튼 = 최소화(hide, 프로세스 유지)**. 단일 인스턴스.
- 트레이 메뉴 v1: ~~`[데몬 시작] [UI 열기(focus or spawn)] [UI 종료(UI만)] [전체 종료]`~~ **→ 6개로 확정(2026-06-18 dashboard7 사용자 결정):** `[데몬 켜기] [데몬 끄기] [UI 열기] [UI 닫기] ──(구분선)── [트레이 종료] [완전 종료]`. 데몬이 메인(영속)·트레이/UI는 클라이언트라는 모델을 메뉴에 반영 — **트레이 종료**=트레이 프로세스만 끔(데몬·UI는 detached로 계속 돌고, 다음 트레이 시작 시 lockfile로 재발견, ADR-0024 C2), **완전 종료**=데몬+UI graceful(C4) 후 트레이까지. 라벨도 켜기/끄기·열기/닫기로 통일. 트레이는 `add(command)` 식 — 항목 추가는 나중에.
- **commit `664d629`(main CloseRequested → `app.exit(0)`) 조정:** X = `hide() + prevent_close()`로 바꾸고, 진짜 종료는 트레이 "전체 종료"에서만. (664d629가 막던 hidden 창 좀비 문제는 "X=종료" 대신 "트레이 종료만 exit"으로 해소.)

## 거부한 대안
- **앱=트레이셸 2프로세스(UI가 트레이 소유, 창 hide):** 교차검증 3종이 "2프로세스가 일반적으로 더 단순"이라 했고 그게 맞다. **그러나 채택 안 함** — 사용자 의도는 "크래시 생존"이 아니라 **로그인 상주 런처 + WebView 비용 없이 트레이만 항상**. 2프로세스는 로그인 때 풀 WebView(무거움)를 항상 띄워야 한다. GPT·Gemini 둘 다 "tray-host가 **순수 Rust(no WebView)** 면 정당, Tauri 앱을 또 띄우는 거면 2프로세스로"라 명시 — 우리는 순수 Rust라 정당 조건 충족.
- **Tauri 앱(WebView)을 tray-host로 분리:** 교차검증이 반대한 바로 그 안. WebView 2벌·메모리·배포 복잡도. 채택 안 함(그래서 tray-host는 순수 Rust).
- **데몬에 트레이 부착(데몬이 GUI 루프 보유):** ADR-0021 windowless 번복 + winit 메인스레드 vs tokio 충돌. 데몬 성격 변경. 거부.

## 근거
- `/consult` 3종(GPT·Gemini·Claude) + 블라인드 judge: (a) "tray-host 별도는 과함 → 단 순수 Rust면 정당"에 합치. 진입점 변경 비용(자동시작·딥링크·단일인스턴스·업데이터)이 크다는 점도 3종 합치 → §영향에 마이그레이션으로 명시.
- 표준 패턴 확인(Discord·Steam): X=트레이 최소화, 트레이 Quit=진짜 종료. amd/gaia가 동일 구조(가벼운 트레이 제어판 + 백그라운드 엔진 + graceful shutdown).

## 영향 / 불변식
- tray-host는 **WebView를 쓰지 않는다**(순수 Rust). 쓰면 2프로세스로 통합하는 게 맞다는 교차검증 결론을 어기는 것.
- UI X 버튼은 **종료가 아니라 hide**. 진짜 종료는 트레이 "전체 종료"만(ADR-0024 순서).
- **진입점이 앱→tray-host로 바뀐다** → 자동시작 등록·단일 인스턴스 가드(✅ 구현 2026-06-19 — 아래 §트레이 싱글 인스턴스)·딥링크(`engram://`)·업데이터·코드사이닝 표면이 전부 영향. 구현 시 마이그레이션 항목으로 다룬다(TRD §7).
- 데몬 소유·생사·종료·데이터 위치는 ADR-0024.

## 트레이 싱글 인스턴스 (2026-06-19 dashboard8 — 구현)
트레이를 여러 번 실행하면 프로세스가 여러 개 떠 시스템 트레이에 아이콘이 쌓인다(실측: 3번 실행=프로세스 3개=아이콘 3개). named mutex `Global\EngramTrayHost-<user>`로 가드 — 두 번째 실행은 기존 인스턴스를 감지해 즉시 `exit(0)` 양보(QA 실측: 2·3번째 exit 0, 최종 1개). 데몬 `instance.rs`와 같은 패턴이되 **이름만 트레이 전용**(데몬 `Global\EngramDashboardDaemon-<user>`와 분리 → 데몬·트레이 동시 기동 가능). env override(`ENGRAM_TRAY_INSTANCE_KEY`)도 데몬(`ENGRAM_INSTANCE_KEY`)과 분리(테스트 격리). 코드는 `crates/engram-tray-host/src/instance.rs`(트레이 자체 보유 — 공용 crate 추출은 안 함, 데몬 instance.rs는 패턴 참고).

**★데몬과 다른 종료 정책(load-bearing — 복사 금지)★:** 가드 생성 실패(`CreateMutexW` 시스템 에러로 NULL 핸들, 드묾) 시 — **데몬은 미기동(exit 1)**이지만 **트레이는 경고 로그 후 강행(가드 없이 띄움)**. 트레이의 단일성은 데이터 정합성이 아니라 **UX(아이콘 중복)** 문제라, 아이콘 부재(데몬·UI 제어 진입점 통째 상실) > 중복 위험. 데몬 정책(단일성=daemon.json/agents.json 단일 발행=정합성, ADR-0008)을 트레이에 그대로 복사하면 안 된다.

거부한 대안: **Err 시 미기동(데몬 정책 복사)** — 트레이는 단일성 손실 비용(아이콘 중복)이 가용성 손실(아이콘 부재)보다 가벼워 가용성 우선이 맞다(reviewer-deep 적출, 사용자 정책과 일치).
