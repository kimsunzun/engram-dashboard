# ADR-0024: 데몬 소유·생사감지·종료·데이터 위치 — self-owned detached + WS/lockfile liveness + 재입양

- 상태: 확정 (2026-06-18, dashboard6 세션 — 사용자 결정 + `/consult` 3종 교차검증 job `20260618-151957-consult-tray-daemon-topology`)
- 관련: ADR-0023(토폴로지)·ADR-0021(on-demand·무재시작)·ADR-0001(kill 2동사·JobObject)·ADR-0008(sid 통제)·CLAUDE.md §5 · `docs/process/S13-tray-lifecycle/tray-topology-trd.md`
- 범위: 데몬을 누가 소유/spawn하나, 생사를 어떻게 감지하나, 전체 종료 순서, 런타임 데이터 저장 위치.

## 맥락
3프로세스(ADR-0023)에서 데몬은 "UI/tray-host가 죽어도 생존"해야 한다. 그러려면 데몬을 누구의 종속 자식으로 두면 안 되고, 생사 감지·종료·고아 회수·데이터 위치를 정해야 한다.

## 결정 (C1~C4 + 데이터)
- **C1 — self-owned, detached.** 데몬은 누구의 OS 자식(특히 JobObject `KILL_ON_JOB_CLOSE`)도 아니다. 누가 ensure하든 spawn은 detached. 싱글톤 lock(lockfile: PID·port·generation/nonce)으로 존재 보증 — 살아있으면 no-op 붙기. 고아 방지는 **idle self-shutdown**(연결 0 ∧ 에이전트 0이 N초 지속, tmux 모델 ADR-0021).
- **C2 — 생사 = WS connection liveness(1차) + lockfile 재발견.** WS heartbeat/종료 이벤트가 1차 신호. tray-host/UI 재시작 시엔 lockfile{PID·port·generation}로 재발견하고, **PID 생존·port 응답·generation 일치를 검사해 stale면 정리 후 fresh spawn**. 부모-자식 exit 신호 단독 의존 금지.
- **C3 — desired-state는 owner(tray-host)가 소유, UI는 직접 ensure 금지.** ~~UI는 연결만. 연결 실패 시 owner에 ensure 요청(또는 owner가 부팅/명령 시 ensure). 현 UI bootstrap의 데몬 ensure 제거.~~ **★폐기 (Superseded by ADR-0025)★** — "UI 켜졌는데 데몬 없음"이 사용자 의도에 모순. UI는 부팅 1회 데몬 ensure를 **유지**한다(데몬 싱글톤이라 tray-host와 UI 양쪽 ensure가 안전). 종료 중 재spawn race는 C4로만 막는다. 상세·근거는 ADR-0025.
- **C4 — 전체 종료 순서:** owner state=Stopping → ensure/open/start 차단 → UI에 full_shutdown(reconnect=false, ensure=false) → 데몬 graceful drain(새 명령 거부 → PTY 자식 정리[ADR-0001 인과] → lock 제거 → exit, 상한 타임아웃 후 Job 강제) → 데몬 exit 확인 → UI 종료 → tray-host 종료.
- **데이터 위치:** `.engram-data/`(`.gitignore` 완료). daemon.json(토큰)·agents.json 등. ~~embedded·daemon 공유(같은 폴더).~~ **영속 유지(휘발 X).**
  - **★데이터 위치·공유 부분 폐기 (Superseded by ADR-0027, 2026-06-21)★:** "embedded·daemon 공유(같은 폴더)" + 아래 "디버그=repo루트/릴리즈=exe옆 수렴" 은 폐기됐다. 모드별 분리로 대체 — **embedded=실행 폴더 `.engram-data`(폴더-로컬), daemon=release는 `%APPDATA%`(유저-global)·dev는 실행 폴더 통일.** 에이전트는 모드 간 비공유. 상세·근거는 ADR-0027. (C1 detached/breakaway·C2 liveness·C4 종료순서·토큰 정책 등 본 ADR 나머지는 유효.)
  - **구현(2026-06-19 dashboard7 확정, sub-step 2):** 경로 주입(런처가 `--data-dir`로 넘김) 대신 **각 프로세스가 self-resolve**(`engram_dashboard_discovery::default_data_dir()` 단일 출처). 우선순위: ① `ENGRAM_DATA_DIR` override > ② **디버그** = `current_exe`에서 위로 올라가 repo 루트(`.git`/`[workspace]`)의 `.engram-data/` > ③ **릴리즈** = exe 폴더 옆 `.engram-data/` > cwd fallback. 세 exe(daemon/embedded/tray-host)가 같은 빌드모드에서 같은 폴더로 수렴(불변식). 주입 안 쓰는 이유: 모든 exe가 `target/<profile>/`에 co-located라 self-resolve가 더 단순하고 주입 배선 0.

- **제어 "허락" 모델:** 협조적 제어=데몬이 lockfile에 발급한 토큰으로 인증(graceful StopDaemon 등). 강제=taskkill 폴백(고아 위험, 폴백 전용). 데몬 WS는 `127.0.0.1` 바인드(로컬 전용) — 외부 노출 0.

## 거부한 대안
- **데몬 = tray-host/앱의 자식 프로세스 + OS wait로 감지(polling 불필요)** — Gemini 제안, judge가 기각. **detached 생존과 모순**: JobObject KILL_ON_JOB_CLOSE에 넣으면 부모 죽을 때 데몬도 죽음(생존 위반). 안 넣으면 부모 핸들이 tray-host 재시작 때 끊겨 어차피 lockfile 재발견 필요 — "polling 불필요"는 행복경로 한정 거짓.
- **UI가 데몬 직접 spawn(현 동작)** — 다중 ensure race + 고아/의도 데몬 구분 모호 + 종료 중 재spawn. C3로 대체.
- **embedded 완전 휘발(파일 0)** — 한때 검토했으나 사용자 거부(복원 유지 원함). embedded도 영속.
- **데이터 = %APPDATA%\com.engram.dashboard 유지** — 정식 앱이면 맞지만(쓰기권한·업데이트 생존) engram은 설치형 미계획 + I:\ 고정 쓰기가능이라 co-located 내부 폴더가 더 가시적·관리 쉬움. `.engram-data/`로.

## 근거
- `/consult` 블라인드 judge: C1/C2의 "detached ↔ 자식소유 모순"을 핵심 적출, Claude안(self-owned + WS liveness)이 가장 정합, GPT안의 재발견 견고성(generation/nonce·stale lock·race 망라) 병합. C4 종료순서·재spawn race 차단은 GPT/Claude 합치(자동재시작 폐기 사유 재발 방지).
- 강제종료 커버(TRD §5b): tray-host kill→데몬 생존→재실행 재입양 / idle self-clean / stale lock 검사 = Discord식 관리불가 좀비 방지.

## 영향 / 불변식
- 데몬 spawn 코드는 **절대 kill-on-close Job에 데몬을 넣지 않는다**(C1). 어기면 부모 죽을 때 데몬 동반 사망 = 생존 깨짐.
- 종료는 **graceful 명령 → 데몬 죽음 확인 → 강제 폴백** 순(C4). fire-and-forget 금지(좀비).
- 토큰은 lockfile에만, 로그 금지(기존 규약 유지). WS는 loopback 바인드 유지. 원격 노출은 미래 별도 보안작업(보안 1급, 사내 검토).
- `.engram-data/`는 git 커밋 금지(`.gitignore` 등록됨).
