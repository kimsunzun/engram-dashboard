# ADR-0028: 백엔드가 이벤트버스를 소유 — 단일 push 채널(백엔드→트레이/WebView/LLM)

- 상태: 확정 (2026-06-21, dashboard10 세션 — 사용자 결정. 트레이 회색-고착 버그를 계기로 개념 일반화)
- 관련: ADR-0003(OutputSink/StatusSink 코어 격리 — 본 ADR이 일반화·확장)·ADR-0024 C2(WS/lockfile liveness)·ADR-0026(트레이 통합)·CLAUDE.md §5(LLM-우선 제어, 손발/두뇌 분리) · `docs/process/step-log.md` dashboard10
- 범위: 백엔드(Rust)의 상태 변화가 어디로/어떻게 흘러 트레이·WebView·(미래)LLM 에 닿나. "기능마다 옵저버/probe 를 따로 다는" 방식을 단일 이벤트버스로 대체한다.

## 맥락
daemon 모드에서 트레이 아이콘이 회색 고착되는 버그가 드러났다(데몬은 실제 alive 인데 트레이는 부팅 시점 1회 probe 로 칠한 회색 그대로). 원인을 파보니 구조 문제였다: **트레이(Rust)가 데몬 생사를 직접 못 보고 daemon.json/PID 를 찍어보는(probe) 수밖에 없어서**다. daemon 모드의 WS 연결(진짜 liveness 신호)을 프론트(WebView JS)가 쥐고 있어 Rust 가 못 보기 때문.

이를 고치려다 "프론트가 트레이로 상태를 위로 push" 안을 냈는데 — 방향이 거꾸로였다. **Rust 가 상위(프로세스·트레이·lifecycle 소유)이고, 이벤트는 백엔드→프론트로 내려가야 한다**(§5: 백엔드=두뇌, 프론트=손발, 죽음 감지도 백엔드 판단). 기능별 bespoke 옵저버·소비자측 probe 는 전부 "단일 push 채널이 없어서" 나온 우회였다.

이미 코드에 이 채널의 **맹아**가 있다: `StatusSink`(core trait, publish 측) → `TauriStatusSink`(`app.emit`) → 프론트 `eventBus.ts`(`agentClient` 구독). `agent-list-updated`/`status-changed`/`restore-result`가 이 길로 Rust→WebView 로 내려간다. 이걸 정식 개념으로 끌어올린다.

## 결정
**백엔드(Rust)가 이벤트버스를 소유한다. 그것이 백엔드 상태 변화의 단일 push 채널이다.**

- **publish 측 = 백엔드.** 모든 백엔드 상태 변화(agent 목록·status·복원 결과·**데몬 생사**·앞으로의 백엔드 로직)는 이 버스에 이벤트로 publish 된다. 기능마다 별도 옵저버/배선을 만들지 않는다 — 새 신호는 버스 이벤트 하나를 추가하는 일.
- **구독자 = 여럿.** ① WebView(Tauri `emit` → 프론트 `eventBus`) ② 트레이(Rust 내부 구독, 네이티브 `set_icon`) ③ (미래)LLM/제어 표면. 같은 버스를 여럿이 구독한다.
- **방향 불변(load-bearing): 백엔드→아래로만.** 상태 push 는 항상 Rust→{트레이, WebView, LLM}. **프론트가 상태를 위로(트레이/Rust) push 하지 않는다.** (프론트→백엔드는 *명령/의도*(command/invoke)지 *상태 push* 가 아니다 — 층위가 다르다.)
- **데몬 생사 = 이 버스의 첫 신규 이벤트.** Rust 가 소유한 생사 옵저버가 변화를 버스에 publish → 트레이(아이콘)·WebView(이벤트) 가 구독해 반영. 옵저버가 *어떻게* 변화를 아느냐는 버스 뒤의 impl 디테일: **pre-flip = Rust 주기 probe(daemon.json/PID, Rust 소유), post-flip = Rust 가 쥔 연결의 connect/disconnect 이벤트(ADR-0024 C2)**. 둘 다 "Rust 소유 → 아래로 push" 형태는 동일 → flip 때 옵저버 내부만 바뀌고 버스·구독자는 그대로(재구축 없음).

### 코어 격리 보존 (ADR-0003 확장, 폐기 아님)
버스의 publish 면은 코어의 sink trait(현 `StatusSink`)으로 유지 — 코어는 Tauri/전송을 모른다. Tauri `emit`·트레이는 **엣지(src-tauri) 구독자/어댑터**다. 즉 0028 은 "sink = 버스의 publish 측, 엣지에서 다중 구독자로 fan-out" 으로 ADR-0003 을 일반화할 뿐, 코어 격리(코어 tauri import 0)는 그대로다.

## 거부한 대안
- **프론트가 상태를 위로 push(프론트→트레이/Rust)** — §5 위반(백엔드=두뇌·죽음 판단 백엔드). 애초에 회색-고착을 만든 구조(연결을 프론트가 쥐어 Rust 가 못 봄)를 *고착*시키는 방향. 또 "나중에 백엔드 로직을 프론트로 내릴 때" 경로가 거꾸로라 안 맞는다. 폐기.
- **기능별 bespoke 옵저버 + 전용 배선** — 새 백엔드→프론트 신호마다 옵저버·이벤트·구독을 새로 짠다. 조합 안 되고, 트레이/프론트가 같은 신호를 따로 받느라 중복·드리프트. 단일 버스로 흡수.
- **소비자측 pull/probe(트레이가 daemon.json probe, 프론트 폴링)** — 단일 출처 없음, stale, 타이밍 버그(회색 고착의 직접 원인). 이게 본 ADR이 제거하는 우회다.
- **주기 폴링을 프론트에 둠** — 폴링 위치가 손발(프론트)이라 거꾸로. 옵저버는 Rust(두뇌) 소유. (폴링 자체는 pre-flip 옵저버의 impl 수단일 뿐 — 버스 뒤에 숨고, flip 때 연결 이벤트로 교체.)

## 근거
- §5(손발/두뇌 분리)·ADR-0003(sink 격리)·ADR-0024 C2(WS liveness)가 이미 가리키던 방향의 명시화. 회색-고착 버그가 "단일 push 채널 부재"라는 공통 뿌리를 드러냈다(사용자 지적).
- 기존 `StatusSink`+`eventBus` 가 동작 중인 맹아라 공수 작음 — 데몬 생사를 같은 패턴의 이벤트로 추가 + 트레이를 신규 구독자로 붙이는 정도.

## 영향 / 불변식
- **트레이 아이콘 색 = 버스의 daemon-liveness 이벤트로만 갱신**(부팅 1회 probe 의존 제거). Rust 옵저버가 push 소스.
- **새 백엔드→프론트 신호는 버스 이벤트로 추가**(전용 배선 금지). "백엔드 로직을 프론트로 내린다" = 버스에 publish.
- 프론트는 버스 **구독만**(상태 위로 push 금지). 프론트→백엔드는 command/invoke(의도)로 한정.
- 코어 tauri import 0 유지(버스 publish 면 = 코어 sink trait, fan-out 은 엣지).
- pre-flip 옵저버 = Rust 주기 probe(버스 뒤 impl), post-flip = 연결 이벤트. 버스·구독자 계약 불변(flip 때 재구축 없음).
