# ADR-0027: 모드별 인스턴스 스코프 + 데이터 위치 — embedded=폴더별/폴더-로컬, daemon=전역/유저-global (ADR-0024 데이터-공유 폐기)

- 상태: 확정 (2026-06-21, dashboard9 세션 — 사용자 결정 + single-instance 패턴 조사 job 서브에이전트)
- 관련: ADR-0026(2프로세스 토폴로지)·ADR-0024(데몬 소유·데이터 — 본 ADR이 데이터 부분 일부 폐기)·ADR-0008(sid 통제)·CLAUDE.md §5 · `docs/process/step-log.md` dashboard9
- 범위: single-instance(중복 실행)를 **무엇 단위로** 잡나, 런타임 데이터(`.engram-data`)를 **어디에** 두나. 둘은 "에이전트를 누가 소유하나"로 묶인 한 결정이다.

## 맥락
ADR-0026으로 2프로세스(embedded 앱 + 별도 daemon)가 됐다. single-instance를 설계하며 "인스턴스 스코프(전역 vs 폴더별)"와 "데이터 위치"가 **같은 결정**임이 드러났다 — 데이터 위치는 에이전트 소유 주체를 따라간다. OSS 조사: 문서/워크스페이스 앱(VS Code·JetBrains·Obsidian)=폴더/프로젝트 단위, 서비스/데몬(Docker·Tailscale)=전역(유저). single-instance 락 키를 "고정 앱 이름"으로 잡으면 전역, "경로 해시"로 잡으면 폴더별.

## 결정
**모드가 다르면 스코프·데이터 위치가 다르다.**

- **embedded(싱글 모드) = 폴더별 + 폴더-로컬 데이터.**
  - 인스턴스 스코프 = **폴더별**. 같은 폴더 재실행 → 막고 기존 창 raise(`show()`+`unminimize()`+`set_focus()`). **다른 폴더 → 각각 독립 인스턴스 허용.**
  - single-instance 락 키 = `{identifier} + hash(작업폴더)`(Tauri 기본 전역 키를 폴더별로 분할).
  - 데이터 = **실행한 폴더의 `.engram-data/`**(그 폴더가 자기 에이전트를 소유).
- **daemon(에이전트 모드) = 전역(per-user) + 유저-global 데이터.**
  - 인스턴스 스코프 = **전역 1개**(기존 named mutex `Global\EngramDashboardDaemon-<user>`).
  - 데이터(release) = **유저 영역 `%APPDATA%\com.engram.dashboard`**(Tauri `app_data_dir` 류). 데몬은 dockerd/tailscaled처럼 per-user 서비스라 폴더에 안 묶임.
- **트레이/X=hide = daemon 모드 전용 (UI 거동도 모드별 — 사용자 결정, B안).** **embedded = 평범한 창 앱**: 트레이 없음, X = 일반 닫기/종료(재오픈 시 cold restore). **daemon = 트레이 상주**(전역 1개), X = hide(트레이로 최소화), 창 닫아도 에이전트 데몬에서 생존. 근거: embedded는 포그라운드 세션(닫으면 끝)·per-folder라 트레이 두면 폴더마다 N개 클러터 + 데몬 메뉴 orphaned. VS Code(폴더별 창·트레이 없음) / Docker Desktop(트레이 상주) 분기와 동형. → ADR-0026의 "X=hide·트레이"는 **daemon 모드로 mode-gate**(스텝3). 현 스텝2 구현(트레이·X=hide 무조건)은 embedded에서 비활성화 필요.
- **에이전트 store는 모드 간 분리(비공유) — 의도된 설계.** embedded(폴더-로컬)에서 만든 에이전트는 daemon(유저-global)에 안 보인다. 다른 스코프라 공유가 오히려 혼란. (사용자 결정: "에이전트 공유되는 게 이상한 거다.")
- **dev(디버그) = 두 모드 모두 "싱글(embedded) 모드 데이터" 참조 (사용자 결정 — #4 해소).** dev에선 daemon도 embedded의 **폴더-로컬 `.engram-data`**(= 현 `default_data_dir` debug 동작: exe에서 repo 루트 walk-up)를 본다. 개발 중 embedded↔daemon **모드 스위칭 테스트**를 같은 로컬 데이터로 하기 위해. **release에서만 daemon이 `%APPDATA%`로 분기**한다. (= dev는 모드 무관 단일 경로라 default_data_dir 모드 인자가 release에서만 갈림.)
- `ENGRAM_DATA_DIR` override(테스트 격리 탈출구)는 그대로 유지 — 최우선.

### 우선순위 (default_data_dir 재설계 방향)
```
1. ENGRAM_DATA_DIR (테스트 격리 — 불변)
2. dev(debug):  실행 폴더의 .engram-data  (embedded·daemon 공통 — 스위칭 테스트)
3. release:
     - embedded:  실행(작업) 폴더의 .engram-data
     - daemon:    %APPDATA%\com.engram.dashboard
```
→ `default_data_dir()`가 **모드(embedded/daemon)를 알아야** 한다(인자 또는 분기). 현재는 모드 무관 단일 함수라 시그니처 변경 필요.

## 거부한 대안
- **embedded·daemon 데이터 공유(ADR-0024 원안: 같은 `.engram-data`)** — embedded는 폴더 종속인데 daemon은 전역이라 한 폴더에 묶으면 daemon의 전역성이 깨진다. 에이전트가 모드 간 새어 보이는 것도 스코프 혼란. → 모드별 분리로 대체.
- **embedded도 전역 단일(옵션1)** — 트레이·데이터 안전엔 단순하나, "프로젝트(폴더)마다 따로 띄워 보는" 사용자 워크플로우와 정면 충돌. 폴더 기준 실행 모델을 버리는 셈. 거부.
- **혼합 데몬 허브(옵션3)를 지금** — 트레이 1개 + 폴더별 병행을 다 잡지만 데몬↔클라이언트 IPC·조정이 무거움. flip 이후 영역. 지금은 embedded 폴더별 + daemon 전역으로 단순하게.

## 영향 / 불변식
- **ADR-0024의 "데이터 위치" 결정 중 "embedded·daemon 공유(같은 폴더)" + "디버그=repo루트 / 릴리즈=exe옆 수렴(세 exe 동일 폴더)"는 폐기** — 본 ADR로 대체. **단 ADR-0024의 나머지(C1 detached/breakaway, C2 WS/lockfile liveness, C4 종료순서, idle self-shutdown, 토큰 lockfile-only)는 유효.**
- single-instance(embedded) 락 키에 **작업폴더 해시** 포함. 같은 폴더 2nd → argv로 1st에 위임 후 raise, 다른 폴더 → 신규 허용.
- **트레이 = daemon 모드 전용(B안).** embedded는 트레이 없음 → 폴더 여러 개여도 트레이 클러터 0(그냥 창 여러 개, VS Code식). daemon은 전역 1개라 트레이도 1개. 스텝2의 무조건 트레이·X=hide는 스텝3에서 daemon 모드로 mode-gate.
- `.engram-data`는 git 커밋 금지(`.gitignore` 유지). appdata 경로는 release-daemon에서만.
- ~~미해결: dev "실행 폴더" 정의~~ **해소(사용자 결정):** dev는 두 모드 모두 **싱글(embedded) 모드 데이터 = 현 debug walk-up `.engram-data` 그대로** 참조(daemon-dev도 동일). release-daemon만 appdata. → default_data_dir 모드 분기는 **release에서만** 필요, dev는 기존 동작 유지(구현 단순).
